# 分页、限流与配额

## 本节目标

学会完整而有边界地读取列表 API，理解 offset/page、cursor 和 link 三类分页；区分限流与配额；遇到 429 时尊重服务端提示，而不是并发重试制造“惊群”。

## 为什么 API 不一次返回全部数据

假设知识库有 100 万条文档。一次返回全部结果会占用服务端内存、网络带宽和客户端内存，也难以在中途失败后恢复。分页把结果拆成多个有限响应。代价是：你必须正确保存下一页位置、处理数据变化，并为循环设置终止边界。

## 三种常见分页

### Page / offset 分页

```http
GET /items?page=3&page_size=50
GET /items?offset=100&limit=50
```

优点是容易跳到指定位置，适合相对稳定的小数据集。若读取过程中前面插入或删除数据，offset 可能造成重复或漏读。服务端对很深的 offset 查询也可能昂贵。

### Cursor 分页

```json
{
  "items": [{"id": "a1"}, {"id": "a2"}],
  "next_cursor": "opaque-value"
}
```

下一次把不透明 cursor 原样传回：

```http
GET /items?cursor=opaque-value&limit=50
```

客户端不应解析或修改 cursor。它通常比 offset 更适合持续变化的数据，但是否保证快照一致、cursor 多久过期，完全取决于 API 契约。

### Link / next URL 分页

服务端直接给出下一页 URL，可能位于 JSON 的 `next` 字段或标准化程度不同的 `Link` header 中。客户端应先验证下一页 URL 是否仍属于预期 host，再携带合适凭据，避免把 Authorization 意外发往不可信主机。

> [!note] 没有统一字段名
> `next_cursor`、`has_more`、`page_token`、`next` 和 `Link` 都可能出现。分页参数与终止条件必须以目标 API 官方文档为准。

## 一个防失控的 cursor 循环

```python
from collections.abc import Iterator

import requests


def iter_items(base_url: str, *, max_pages: int = 100) -> Iterator[dict]:
    cursor: str | None = None
    seen_cursors: set[str] = set()

    with requests.Session() as session:
        for _ in range(max_pages):
            params = {"limit": 50}
            if cursor is not None:
                params["cursor"] = cursor

            response = session.get(
                f"{base_url.rstrip('/')}/items",
                params=params,
                headers={"Accept": "application/json"},
                timeout=(3.05, 15),
            )
            response.raise_for_status()
            page = response.json()

            items = page.get("items")
            if not isinstance(items, list):
                raise ValueError("响应缺少 items 数组")
            yield from items

            next_cursor = page.get("next_cursor")
            if next_cursor is None:
                return
            if not isinstance(next_cursor, str) or not next_cursor:
                raise ValueError("next_cursor 格式错误")
            if next_cursor in seen_cursors:
                raise RuntimeError("服务端返回了重复 cursor，停止循环")

            seen_cursors.add(next_cursor)
            cursor = next_cursor

    raise RuntimeError(f"超过最大页数 {max_pages}")
```

关键保护：

- `max_pages` 防止契约错误造成无限循环；
- 记录已见 cursor，发现重复立即停止；
- 校验 `items` 与 cursor 类型；
- 使用生成器逐条消费，避免先把全部页装进内存；
- 若任务要断点续传，将**下一页 cursor** 和处理进度安全持久化。

即使 cursor 分页也可能返回重复记录。对导入任务可基于稳定资源 ID 去重，并明确“重复跳过”与“更新已有记录”的业务规则。

## 限流与配额不是一回事

- **限流（rate limit）**：限制短时间内请求速率或并发，例如每分钟请求数。
- **配额（quota）**：限制一个较长周期的累计用量，例如每月请求量、token 或费用预算。
- **并发限制**：同一时间允许的在途请求数。

不同 API 可能按账户、项目、模型、用户、IP 或 endpoint 分桶。响应 header 名和计数算法没有统一约定，不能假设一定存在 `X-RateLimit-Remaining`。

## 处理 429

RFC 6585 定义 `429 Too Many Requests`，响应可能带 `Retry-After`。该 header 可能是秒数，也可能是 HTTP 日期：

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 30
```

客户端策略：

1. 优先读取目标 API 的限流文档；
2. 若有合法 `Retry-After` 且未超过本次等待预算，等待完整建议时间；若它超出 deadline/等待预算，停止当前重试并交给外层延期，不能截短后提前请求；
3. 若没有，使用带 jitter 的指数退避；
4. 降低全局请求速率或并发，而不是让每个 worker 独立猛试；
5. 设置最大尝试次数/总重试预算，耗尽后把任务转为可恢复失败；
6. 记录状态码、endpoint 模板、尝试次数、等待时间和 request ID，不记录 token。

如果 429 是“月度配额已用尽”，等几秒重试没有意义。机器错误码、重置时间和管理后台指标能帮助区分短期限流与长期配额。

## 主动限速

等到 429 才减速会浪费请求。批处理可采用：

- 固定 worker 数限制并发；
- 令牌桶或漏桶控制全局速率；
- 根据服务端剩余额度动态减速（只有文档明确 header 语义时）；
- 把交互请求与后台批处理放入不同队列；
- 对同一资源合并重复请求，使用缓存或批量 endpoint。

Agent 循环尤其要防止模型反复选择同一工具。API 客户端的限速只能缓解后果，Agent 层还应限制单次任务的工具调用次数和总预算。

## 分页与限流组合时的恢复

每成功处理一页后再保存 cursor。若在下一页遇到 429 或临时 5xx，重试当前页；若进程重启，从已保存 cursor 继续。写入下游系统时还要保证重复处理安全，例如用源资源 ID 做 upsert。

```text
获取一页 -> 校验 -> 写入/处理 -> 保存 next_cursor -> 获取下一页
                         ^ 成功后才推进检查点
```

## 常见错误

- 看到 `items` 为空就总是终止：有的契约以 `next_cursor` 为准；按文档确定。
- cursor 重复仍继续循环。
- 先收集所有页再处理，导致内存峰值和失败后全部重来。
- 每个线程各自重试 429，瞬间形成更多请求。
- 将所有 429 当作短期速率限制，忽略配额耗尽。
- 日志打印完整 next URL，而该 URL 可能包含敏感签名 query；只记录安全字段。
