# 认证、状态码与凭据安全

## 本节目标

理解“你是谁”和“你能做什么”是两个问题；会按文档使用 API key、Basic Auth 和 Bearer token；能根据常见状态码采取正确动作，并避免把凭据写进代码、URL、日志或仓库。

## Authentication 与 Authorization

- **认证（authentication）**：证明调用方是谁，例如验证 API key 或 access token。
- **授权（authorization）**：判断这个身份能否读取某资源或执行某动作，常由 role、scope、组织和资源归属决定。

一个 token 有效但 scope 不足，通常属于授权问题，而不是“重新登录就必然解决”。

## 四类常见机制

### API key

API key 通常标识项目、应用或账户。具体 header 名由厂商定义，常见形式包括：

```python
headers = {"X-API-Key": api_key}
```

有些服务要求 `Authorization: Bearer <api-key>`；即使内容叫 API key，也必须按该服务文档传递，不能自行改 header 名。API key 适合服务端到服务端调用，但通常不能表达“某位用户委托应用访问资源”的完整授权流程。

### HTTP Basic Auth

Basic Auth 把用户名和密码组合后编码传输；编码不是加密，因此必须使用 HTTPS。Requests 用 `auth=` 构造，不要自己拼 Base64：

```python
import requests

response = requests.get(
    "https://example.com/protected",
    auth=("username", "password"),
    timeout=(3.05, 15),
)
```

### Bearer token

Bearer 的含义是“持有者即可使用”。RFC 6750 定义了常见 header 形式：

```python
headers = {"Authorization": f"Bearer {access_token}"}
```

泄漏 token 的人可能直接冒用，因此需要 HTTPS、最小 scope、较短有效期、安全存储和轮换。不要把 bearer token 放在 query string 中；URL 更容易进入访问日志、历史记录或监控数据。

### OAuth 2.0

OAuth 2.0 是授权框架，不等同于某一种 token 格式。典型参与者包括资源所有者、客户端、授权服务器和资源服务器。客户端先按授权流程取得 access token，再用它访问资源；refresh token（若颁发）只用于向授权服务器换取新 access token，不应发给普通资源 API。

初学阶段先掌握：

- access token 有 scope、期限和目标资源限制；
- 用户委托场景和机器自身调用会使用不同授权流程；
- 不要手写 OAuth 重定向、安全校验和 token 刷新细节，优先使用服务官方 SDK 或成熟 OAuth 库；
- OAuth 安全建议会更新，实施时同时查阅服务文档与 RFC 9700 等当前 BCP。

## 在 Windows 中管理本地开发凭据

代码只读取环境变量：

```python
import os

api_key = os.environ.get("EXAMPLE_API_KEY")
if not api_key:
    raise RuntimeError("缺少 EXAMPLE_API_KEY")
```

当前 PowerShell 进程临时设置占位值：

```powershell
$env:EXAMPLE_API_KEY = "replace-with-local-secret"
python .\app.py
```

关闭终端后该进程变量消失。若项目采用 `.env`，只提交 `.env.example`，将 `.env` 加入 `.gitignore`，并确认未被 Git 既有历史跟踪。生产环境应使用部署平台的 secret manager，而不是把 token 烘焙进镜像或配置文件。

> [!danger] 发生凭据泄漏时
> 从文件中删除字符串并不等于风险解除。应停止继续传播，按提供方流程撤销或轮换凭据，并在获得明确授权后处理 Git 历史；不要在聊天、日志或截图中再次粘贴原值。

## 状态码是决策输入

| 状态码 | 常见含义 | 客户端首要动作 |
| --- | --- | --- |
| `200 OK` | 成功并带响应 | 校验 content type 与数据结构 |
| `201 Created` | 已创建资源 | 读取 body 或 `Location`（若提供） |
| `202 Accepted` | 已接收、可能异步处理 | 按文档轮询任务或等待回调 |
| `204 No Content` | 成功但没有 body | 不要调用 `.json()` |
| `301/302/307/308` | 重定向 | 检查是否预期；敏感 header 不应被错误转发到其他主机 |
| `400 Bad Request` | 请求语法或参数有误 | 修正请求，不盲目重试 |
| `401 Unauthorized` | 缺少或不接受认证凭据 | 检查 token 是否存在、有效、发给正确服务；按流程刷新 |
| `403 Forbidden` | 身份已知但无权访问，或策略拒绝 | 检查 scope、角色、资源归属与策略 |
| `404 Not Found` | 资源不存在，或服务刻意隐藏其存在 | 检查 ID、路径、版本和权限 |
| `409 Conflict` | 与当前资源状态冲突 | 读取错误码；可能是版本冲突或重复操作 |
| `422 Unprocessable Content` | 语法可读但字段语义/校验失败 | 根据字段错误修正 payload |
| `429 Too Many Requests` | 触发限流 | 尊重 `Retry-After`，降低速率/并发 |
| `500 Internal Server Error` | 服务端意外失败 | 有条件地重试并记录 request ID |
| `502/503/504` | 网关、不可用或超时 | 通常是临时故障候选，但仍受重试预算与幂等性约束 |

同一个状态码在不同 API 的业务细节不同。客户端应同时读取文档化的机器错误码，例如 `invalid_scope`、`rate_limit_exceeded`，不要解析可能翻译或改写的自然语言 `message`。

### 401 与 403 的排查顺序

1. 请求是否发到正确 host 和环境（测试/生产）？
2. 是否按文档使用正确 header 和 scheme？
3. token 是否过期、撤销或属于另一个项目？
4. token 的 scope、角色、组织和资源归属是否匹配？
5. 服务端是否返回 `WWW-Authenticate` 或机器错误码？

不要把完整 token 打印出来“确认是否读到”。只记录布尔状态，例如 `credential_configured=true`，必要时记录本地配置来源的名称而非值。

## 一个安全的请求骨架

```python
import os
import requests


def fetch_profile(base_url: str) -> dict:
    token = os.environ.get("EXAMPLE_ACCESS_TOKEN")
    if not token:
        raise RuntimeError("未配置 EXAMPLE_ACCESS_TOKEN")

    response = requests.get(
        f"{base_url.rstrip('/')}/v1/profile",
        headers={
            "Accept": "application/json",
            "Authorization": f"Bearer {token}",
            "User-Agent": "agent-learning-client/0.1",
        },
        timeout=(3.05, 15),
    )
    response.raise_for_status()
    return response.json()
```

这段代码尚未处理重试、schema 与错误 body；它的重点是：凭据从环境读取、只走文档规定的 header、设置超时、没有日志泄漏。

## 常见错误

- 把 API key 放进示例、Markdown、notebook 输出或异常消息。
- 遇到 401 就无限刷新 token；刷新失败也需要终止和报警。
- 将 403 当成服务故障重试；权限不会因等待几秒自动出现。
- 为“调试方便”关闭 TLS 证书验证。
- 将生产 token 用在本地演示或测试数据中。
- 捕获 HTTPError 后只打印 `str(exc)`，丢掉状态码、request ID 和安全可记录的错误码。
