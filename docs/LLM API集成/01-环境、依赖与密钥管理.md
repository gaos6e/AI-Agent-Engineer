# 环境、依赖与密钥管理

## 本节目标

在 Windows 11 与 PowerShell 7 中建立隔离环境，理解依赖固定与密钥边界，不把真实凭据写进代码、笔记或日志。

## 先用 venv + pip

```powershell
$lab = Join-Path $HOME "projects\llm-api-lab"
New-Item -ItemType Directory -Path $lab -Force | Out-Null
Set-Location $lab
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
```

这里刻意把实验目录放在 vault 外，避免提交 `.venv`、缓存或 SDK 产物。随后根据供应商**当前官方 Python SDK 文档**安装对应包，并记录经过测试的版本。课程不硬编码包版本，因为它们会变化。团队项目应使用锁文件或带版本范围的 requirements，并在升级 SDK 前跑 adapter、流式事件和错误映射回归。熟悉 `venv + pip` 后可评估 `uv` 等工具，但它们不改变隔离、锁定和验证原则。

## 密钥只从安全边界进入

当前 PowerShell 会话可使用占位值演示：

```powershell
$env:LLM_API_KEY = "replace-with-a-local-secret"
```

Python 只检查变量是否存在，不打印值：

```python
import os

api_key = os.environ.get("LLM_API_KEY")
if not api_key:
    raise RuntimeError("LLM_API_KEY is not configured")
```

生产环境使用平台的 secret manager，并为开发、测试、生产使用不同凭据与最小权限。不要将密钥作为 URL 查询参数，不要放进 `.env.example`、异常文本、截图、Notebook 输出或完整 HTTP 日志。若使用本地 `.env`，它必须被忽略；示例文件只能写变量名和占位符。

环境变量只解决“如何传入”，不自动解决权限、轮换和泄露。密钥应限制到需要的项目/环境，设置轮换和吊销流程；发现密钥进入 Git 历史后，应停止继续使用并按组织流程轮换，不能只补 `.gitignore`。

## 配置分层

- 非秘密配置：供应商标识、逻辑模型别名、超时、最大尝试次数，可进入版本控制。
- 秘密：API key、临时令牌，只来自环境或密钥服务。
- 动态限制：价格、速率、模型可用性，从官方文档或管理 API核对，不散落在代码常量里。

启动时验证配置，日志仅报告字段是否存在和使用的非秘密配置 ID。

## 练习与自测

创建一个新 venv，写脚本只检查 `LLM_API_KEY` 是否配置并输出布尔值。关闭终端后确认会话级环境变量不再存在。自测：你的错误追踪系统会自动收集环境变量或请求头吗？若会，应如何过滤？

## 掌握检查

- [ ] 虚拟环境位于 vault 外，依赖版本可复现且升级前有回归测试。
- [ ] 密钥只来自环境或 secret manager，代码、`.env.example`、日志和截图中没有真实值。
- [ ] 开发、测试和生产凭据隔离，并有最小权限、轮换与吊销流程。
- [ ] 启动失败只报告缺少哪个配置，不回显凭据或认证头。
- [ ] 我先掌握 `venv + pip`，再引入 `uv` 等工具，并保留相同安全边界。
