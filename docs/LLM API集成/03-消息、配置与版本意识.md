# 消息、配置与版本意识

## 本节目标

设计稳定的应用请求合同，把角色、内容、模型选择和供应商特性留在明确的适配边界。

## 供应商无关合同

应用层可以先定义自己真正需要的最小字段：

```json
{
  "operation_id": "ticket-42-attempt-group-1",
  "prompt_version": "ticket-router-1.3.0",
  "model_profile": "fast-structured-classifier",
  "messages": [
    {"role": "user", "content": "……"}
  ],
  "output_schema_id": "ticket-label-1.1.0"
}
```

`model_profile` 是内部逻辑配置，不是供应商模型名。部署配置再将它映射为供应商、模型标识、输出上限和受支持功能。这样可以审计和切换，但不能假装所有模型的参数语义一致。

内部合同还应区分“必需能力”和“偏好”。例如 schema 严格遵循、工具调用或图片输入若是正确性的前提，就必须在发送前做 capability check；延迟档位或缓存则可能只是优化。adapter 遇到必需能力不支持时返回 `unsupported_capability`，不能静默降级为纯文本再让下游误以为合同仍成立。

## 角色与内容映射

不同 API 对系统/开发者/用户/助手角色、内容块、图片、工具结果和对话状态有不同约束。adapter 应做显式映射并拒绝无法表达的功能，而不是静默降级。高层指令是否跨响应链继承、历史如何计费、服务端状态保留多久，都属于动态事实。

## 配置要可追溯

记录：

- 供应商 adapter 版本；
- 逻辑模型配置 ID 与实际模型标识；
- 提示、schema、工具定义和上下文选择器版本；
- 关键生成参数及其“未设置/显式设置”状态；
- 请求发起时间和应用 operation ID。

不要从某个供应商复制 `temperature`、`top_p` 或推理参数到所有模型。先查支持情况，再用任务评测证明设置的价值。模型别名可能漂移；需要强复现时使用供应商提供的稳定版本机制并保留上线评测。

配置记录要同时保留“请求的模型配置”和“响应实际报告的模型/版本”。别名、路由或后端更新可能让两者不同；只有内部 `model_profile` 而没有实际响应标识，无法解释回归。

## 输出上限和停止状态

输出达到上限、被安全策略拒绝、工具调用待处理和正常完成不是同一种成功。adapter 应把供应商停止原因映射为有限的内部状态，同时保留原始类型，未知新类型要显式暴露而不是当成完成。

## 练习与自测

为“结构化工单分类”设计一个最小内部请求/响应 dataclass。列出换供应商时必须适配的字段。自测：若新供应商不支持所需 schema 子集，系统会报能力不匹配，还是悄悄改用纯文本？

## 掌握检查

- [ ] 应用合同只包含业务需要，供应商字段由 adapter 显式映射。
- [ ] 必需能力与性能偏好分开；无法表达的必需能力会失败关闭。
- [ ] 请求和响应都保留实际模型、adapter、提示、schema、工具和上下文版本。
- [ ] 未设置与显式设置参数可区分，不把某家供应商参数机械复制到另一家。
- [ ] 完成、拒绝、截断、等待工具和未知终止原因映射为不同内部状态。
