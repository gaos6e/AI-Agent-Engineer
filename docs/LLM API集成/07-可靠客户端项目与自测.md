# 可靠客户端项目与自测

## 项目目标与边界

在完全离线环境实现一个供应商无关的可靠客户端内核：严格请求/响应合同、暂时与永久错误、`Retry-After`、指数退避、抖动、attempt 与 retry deadline 双上限，以及 canonical 流事件状态机。mock transport 让失败顺序可重复，不需要 SDK、网络或密钥。

这不是一个真实 HTTP 客户端。它不替代 SDK 的连接/读取/流空闲 timeout，也不声称 canonical 事件名属于任何供应商。真实 adapter 负责把官方 SDK 异常、原始事件、usage 和 request ID 映射到本地合同，并单独接受集成测试。

## 运行环境

脚本只使用 Python 标准库。在 Windows 11 的 PowerShell 7 中执行：

```powershell
cd "X:\path\to\your-vault\Knowledge\AI Agent Engineer\docs\LLM API集成"
py -3.11 --version
py -3.11 -B -m unittest -v .\examples\test_reliable_client.py
py -3.11 -O -B -m unittest -v .\examples\test_reliable_client.py
```

`-B` 禁止写入 `.pyc`。因为项目零依赖，验证时不在 vault 内创建 `.venv`；真实 adapter 项目应按 [[LLM API集成/01-环境、依赖与密钥管理|环境、依赖与密钥管理]] 在 vault 外使用 `venv + pip` 并固定 SDK 版本。

## 分步骤阅读与实验

### 1. 检查 canonical 合同

`LLMRequest` 保存稳定的 `operation_id`、提示版本、逻辑模型配置与输入；同一业务操作的所有 attempts 复用同一请求对象。`LLMResponse` 把 `completed`、`refused`、`truncated` 分开，并保留服务端 request ID、供应商、实际模型和原始用量单位。

这些类只验证本地字段形状。它们不会自动脱敏 `user_text`；生产日志必须使用 allowlist，不能序列化整个对象。

### 2. 验证重试只处理已分类暂时错误

`call_with_retry()` 只捕获 `TransientError`。认证、权限、无效请求、额度耗尽与能力不支持属于 `PermanentError`；adapter bug 等未知异常也直接暴露，不会因为“可能恢复”而盲目重试。

退避等待同时考虑：

- 本地指数上限与有界抖动；
- adapter 解析出的 `retry_after_seconds` 最小等待；
- `max_attempts`；
- `retry_deadline_seconds`。

测试注入假时钟、假 sleeper 和固定随机数，所以不会真实等待。`retry_deadline_seconds` 约束的是本层重试调度；真实 SDK 仍必须设置 per-attempt 的连接、读取/流空闲和单次调用 timeout。

### 3. 观察 attempt 证据

成功结果携带每次 attempt 的 outcome、结构化错误类别、供应商 request ID 和计划等待。`operation_id` 保持稳定，request ID 可以每次不同。耗尽时 `RetryExhaustedError` 区分 `attempts_exhausted` 与 `retry_deadline_exhausted`，并保留最后一个暂时错误。

这仍不自动保证外部副作用幂等。发信、扣费、写库或执行工具应在完整生成与校验之后，由受信任业务层使用自己的唯一键提交。

### 4. 验证流式状态机

本地 canonical 协议只有四类：`response.started`、`response.text.delta`、`response.finished` 与 `response.failed`。只有一个合法 `finished` 才返回 `StreamResult`；拒绝和截断作为显式 terminal status 返回，不等同于完成。

流中 failure 或连接正常结束但缺少 terminal event 时，异常保留 request ID 与 partial text，并把 partial 标记为失败状态。未知 canonical 事件、字段漂移、顺序错误、非法 usage 和终止后继续到来的事件都会失败。真实 adapter 可忽略已明确理解的供应商遥测事件，但必须先按当前官方协议作出显式决定。

## 本机验证记录

2026-07-14 在 Python 3.11.9 上完成：

- `py -3.11 -m py_compile`：脚本与测试通过。
- 普通模式：20 项测试全部通过，warnings-as-errors。
- `python -O` 模式：同一 20 项测试全部通过，关键门禁不依赖会被移除的裸 `assert`。
- 测试不发网络请求、不读取环境变量、不创建真实等待，也不需要 API key。

未执行任何供应商 live API、SDK 兼容性、真实 timeout、限流头或流式事件测试，也未验证 Obsidian 阅读视图。

## 接入真实供应商的步骤

1. 固定官方 SDK 版本，核对当前认证、timeout、默认重试、错误、限流、streaming、usage 和 schema 文档。
2. 新建 adapter，把原始异常按结构化 code/type 映射；不要依赖错误消息字符串。
3. 明确由 SDK 还是本层拥有重试，关闭或预算化其他层，避免请求数相乘。
4. 从 secret manager 或环境变量读取密钥，不输出值；用最小非敏感请求做 smoke test。
5. 将真实 typed events 映射到 canonical 状态，故障注入流中断、未知事件、拒绝、截断和不完整工具参数。
6. 记录 operation/trace ID、每次 request ID、版本、usage、延迟与结果状态；不默认记录原始正文。
7. 跑任务评测、受控集成测试和小流量验证后再上线。

## 扩展任务

1. 为一个实际 SDK 增加 adapter，但保留离线测试完全不联网。
2. 注入 fake transport latency，增加 per-attempt timeout 与端到端业务 deadline 测试。
3. 解析 RFC 9110 的 `Retry-After` delta-seconds/HTTP-date，再映射到 canonical 秒数。
4. 增加本地并发/令牌桶调度，比较“提前排队”与“收到 429 后重试”的故障率。
5. 为工具调用增加业务幂等键和状态存储，模拟“服务端已执行但响应丢失”。

## 自测题

1. 为什么相同 429 可能分别属于暂时限流和永久额度问题？
2. SDK 两次、外层三次、队列四次重试时最坏会有多少 attempt？
3. retry deadline 与一次 SDK read timeout 分别约束什么？
4. HTTP 200 和首个 text delta 为什么都不能证明流已完成？
5. 为什么中断后不能把旧 partial 与新生成直接拼接？
6. operation ID、trace ID 与供应商 request ID 各自解决什么问题？

## 掌握检查

- [ ] 我能运行普通与 `-O` 两组 20 项测试，并解释每类失败。
- [ ] 只有已分类暂时错误进入一个有 attempts、deadline、退避和抖动的重试预算。
- [ ] 永久与未知错误不重试；额度不足不会被当成普通 429 循环。
- [ ] 完成、拒绝、截断、流中错误、缺少终止和未知事件各有明确状态。
- [ ] partial 不会被当成完整输出，外部副作用在完整校验后且具备业务幂等性。
- [ ] 真实 adapter 的 SDK 版本、默认重试、timeout、事件与 usage 均由官方资料和集成测试证明。
