# Agent 核心

## 知识库简介

Agent 不是“让模型一直思考”，而是模型决策与确定性 runtime 共同组成的受控系统：

```text
目标 + 状态 + 观察
→ 模型提出结构化下一动作
→ runtime 校验策略、权限、预算和审批
→ 工具改变/读取环境
→ observation 回写状态
→ verifier 决定继续、等待或终止
```

本知识库从零建立单 Agent 的供应商无关核心：loop、Agent/工作流边界、状态/context/memory、计划与终止、检查点与幂等、人在回路、不可信 observation，以及一个可运行的离线 runtime。

> [!info] 动态事实边界
> 资料获取/复核日期为 2026-07-21。各家 Agent SDK、托管 session、审批、memory 与 tracing API 会变化；课程主体只依赖稳定工程边界。产品接入时重新核对官方 SDK 文档、版本和双方实际行为。NIST AI RMF 1.0 官方页面当日仍注明正在修订；本库引用其通用角色/监督原则，不把它当行业合规意见。

## 学习目标

- 实现 model proposal 与 runtime execution 分离的 observe–decide–act loop。
- 根据路径不确定性、验证、风险和成本选择单次调用、workflow 或 Agent。
- 区分权威 state、event log、context、工作摘要与长期 memory。
- 让计划项、progress、成功证据和所有终止状态可审计。
- 识别 crash window，用 checkpoint、idempotency key、intent digest 和 receipt 恢复。
- 把 clarification、approval、rejection、takeover 和 cancel 做成持久状态。
- 把所有 tool/resource 内容视为不可信 observation。
- 用 deterministic policy 和离线工具测试 runtime，不把模型偶然表现当安全证据。

## 核心责任图

| 组件 | 可以做 | 不能单独决定 |
| --- | --- | --- |
| model/policy | 提议 action、ask、finish candidate | 权限、审批、真正 completed |
| runtime/harness | 校验、预算、执行、状态迁移、终止 | 代替用户定义业务目标 |
| tool adapter | 调用外部系统、规范化 observation | 提升自身权限/信任 |
| state/event store | 保存事实、checkpoint、审计 | 把摘要变成审批事实 |
| verifier | 检查测试、receipt、外部状态 | 绕过失败/负向标准 |
| human | 澄清、批准、拒绝、接管 | 在信息不足时被迫“橡皮图章” |

## 动手实践入口

项目使用 Python 标准库，不联网、不调用真实模型、不需要 key：

以下命令从同时包含 `docs/` 与 `.website/` 的项目根目录运行：

```powershell
Push-Location -LiteralPath 'docs\Agent 核心' # 临时进入本章节目录，之后会恢复原工作目录
python -B .\examples\bounded_agent.py # 运行离线 demo；-B 防止生成 __pycache__ 文件
python -B .\examples\test_bounded_agent.py # 用普通解释器运行全部回归测试
python -B -O .\examples\test_bounded_agent.py # 在优化模式下验证关键控制没有依赖 bare assert
python -B -W error .\examples\test_bounded_agent.py # 将任何 Python 警告升级为测试失败
python -B -O -W error .\examples\test_bounded_agent.py # 同时启用优化与“警告即错误”两项更严格检查
Pop-Location # 返回执行命令前所在的目录，避免影响后续终端操作
```

当前基线：

- demo：恶意 observation → 写操作暂停 → checkpoint → 审批 → commit 后模拟 crash → receipt 恢复 → completed；
- 已满足路径：查询确认 ticket 已关闭 → 不提议写操作、不请求审批 → `already_satisfied` 完成；
- close side effect 恰好 1 次；
- 68 个测试；
- 普通、`-O`、`-W error` 和 `-O -W error` 四种模式全部通过；
- 0 网络、0 第三方依赖、0 凭据。

## 掌握标准

- [ ] 能说明 model/policy 提议动作，runtime 才拥有执行控制权。
- [ ] 能为真实任务比较确定程序、单次调用、workflow 和 Agent。
- [ ] 能从 state/event 重建下一步，不依赖完整聊天历史。
- [ ] 能解释摘要为何不能覆盖 approval/state。
- [ ] 能定义 progress、completion evidence、waiting/terminal 状态。
- [ ] 能画出四个 crash window。
- [ ] 能设计同 key/同 intent 缓存与同 key/异 intent 冲突。
- [ ] 能设计绑定 fingerprint、state version、scope 和 expiry 的 approval。
- [ ] 能让恶意 observation 即使劫持模型也无法越权。
- [ ] 能运行并解释 68 tests 和项目限制。
- [ ] 能明确何时应退回固定 workflow。
