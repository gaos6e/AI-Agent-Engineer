# 执行循环与调用关联

## 本节目标

- 建立供应商无关的工具调用状态机；
- 正确关联每个 call 与 result；
- 区分完成、需工具、需审批、拒绝、截断和故障；
- 用预算、重复检测和进度条件阻止无限循环。

## 最小五步循环

OpenAI 当前官方文档把客户端 function calling 概括为五步：

1. 请求模型并提供工具；
2. 接收模型的 tool call；
3. 应用侧执行代码；
4. 携带对应 tool output 再请求模型；
5. 接收最终回答或更多 tool calls。

Anthropic 客户端工具同样需要应用驱动循环，只是消息块和 stop reason 不同。Gemini 也明确模型不会替应用执行函数。

## 用内部状态机隔离供应商差异

不要在业务代码到处判断 `function_call` 或 `tool_use`。Adapter 把它们归一为有限状态：

```text
MODEL_REQUESTED
  ├─ FINAL_TEXT ─────────────→ COMPLETED
  ├─ TOOL_PROPOSALS ─────────→ VALIDATING
  │     ├─ approval needed ──→ WAITING_APPROVAL
  │     ├─ executable ───────→ EXECUTING → RESULTS_READY
  │     └─ invalid/denied ───→ RESULTS_READY(error envelope)
  ├─ REFUSAL ────────────────→ REFUSED
  ├─ TRUNCATED ──────────────→ INCOMPLETE
  └─ API_FAILURE ────────────→ FAILED/RETRY
```

`RESULTS_READY` 再送回模型，可能回到 `TOOL_PROPOSALS`。只有明确完成原因才是成功；“没有 tool call”可能是拒绝、max tokens 或服务错误。

## 三类 ID

| ID | 范围 | 作用 |
| --- | --- | --- |
| Provider response/item ID | 供应商响应 | 排障与继续 API 状态 |
| call ID | 一次模型提出的调用 | 将 tool result 对回正确 call |
| operation ID | 整个业务任务 | 跨模型回合、服务和审批追踪 |
| idempotency key | 业务意图 | 防止重试产生重复副作用 |
| approval ID | 人类确认记录 | 审计谁批准了哪项动作 |

Call ID 不能代替幂等键：模型重试时可能产生新 call ID，却仍是同一次退款意图。

## Provider Adapter 的职责

Adapter 应：

- 解析零、一个或多个 call；
- 保留 call ID、name 与原始完成原因；
- 严格解析 arguments；
- 把内部 result 变成供应商要求的回传形状；
- 处理流式参数的完整性与 done 事件；
- 携带供应商要求延续的对话/推理项；
- 把 API 错误映射到内部有限状态；
- 记录 provider、API、SDK、model 与 adapter revision。

Adapter 不应：

- 决定用户是否有权退款；
- 自动修正金额/路径并执行；
- 直接调用任意模型给出的函数名；
- 把供应商错误文本作为业务状态。

## Registry 与 Dispatcher

```python
registry = {
    "get_order": ToolSpec(
        schema_version="get-order-v1",
        risk="read",
        timeout_ms=500,
        handler=get_order,
    )
}
```

Dispatcher 从 registry 获取 schema、风险、超时、授权策略和 handler。未知名称拒绝；禁止：

```python
# 危险示意：不要这样做
handler = globals()[model_name]
handler(**model_arguments)
```

显式 registry 使能力暴露面可审计、可测试、可按用户/任务裁剪。

## 每轮算法

```text
deadline / budgets 初始化
while 未终止:
    调模型
    解析完成原因与 calls
    若 final → 校验并结束
    若 refusal/truncated/error → 对应状态结束或受控恢复
    对 calls:
        去重 call ID
        schema + auth + policy
        若需要审批 → 持久化 pause state
        否则执行并产生严格 result
    关联所有结果并回传
    更新进度、成本与重复轨迹
```

需要审批时不要占用进程等待。保存 operation、conversation continuation、call、参数摘要、策略版本与 expiry；用户批准后恢复并重新校验。

## 循环保护

至少设置：

- 最大模型回合；
- 最大工具调用数；
- 每工具与总 deadline；
- token/费用预算；
- 最大并行度；
- 重复 `tool + canonical arguments` 检测；
- 无进度计数；
- 结果大小上限；
- 人工升级/安全停止路径。

若模型连续调用相同工具与参数却没有新信息，返回结构化 `NO_PROGRESS` 或转人工，而不是不断把相同结果喂回去。

## 流式调用

流式 API 可能分片发送工具参数。只能在供应商明确的完成事件后解析并执行：

- 片段按 item/output index 关联；
- 限制累计字节；
- JSON 完整前不运行 handler；
- 流中断视为 incomplete，不“补一个括号”后执行；
- 最终解析仍走 schema、授权和审批。

进度 UI 可以展示“模型正在准备调用 X”，但不能显示未完成参数为已执行。

## 动手实践

画出“查订单 → 创建退款草稿 → 等待批准 → 提交退款 → 最终说明”的状态机，标出：

1. call ID、operation ID、approval ID、idempotency key；
2. API max tokens；
3. 未知工具；
4. 审批过期；
5. 提交超时但下游可能已成功；
6. 模型再次提出完全相同调用。

为每个分支写终止/恢复状态，不能只写“报错”。

## 常见错误

- `stop_reason != tool_use` 就当成功；
- 多个 call 共用一个无标识结果字符串；
- 只保存 call ID，不保存业务 operation；
- 流式 arguments 未完成就执行；
- 审批等待只存在进程内存；
- 无限循环没有 deadline/进度条件；
- Provider message 形状直接渗入 handler。
