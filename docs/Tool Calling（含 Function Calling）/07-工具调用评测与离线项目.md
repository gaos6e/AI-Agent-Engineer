# 工具调用评测与离线项目

## 项目目标

运行一个不连接真实模型、网络或业务服务的 trusted-boundary dispatcher，验证：

```text
模型候选 call
  → registry allowlist
  → exact arguments / enum
  → trusted principal + tenant/resource authorization
  → call ID conflict
  → idempotency conflict/cache
  → bound and expiring approval
  → simulated execution/failure
  → strict untrusted result envelope
```

它验证的是应用执行合同，不验证模型能否正确选工具或填参数。真实模型路由需要另建 eval。

## 环境与运行

- Windows 11、PowerShell 7；
- Python 3.11 已验证；
- 仅标准库，无需安装包；
- 不需要 API key、网络、数据库或模型；
- 使用 `-B` 避免 `__pycache__`。

从 vault 根目录运行全部 16 个场景：


## 两个 Mock 工具

### get_order

- 只读；
- 参数只有 `order_ref`；
- tenant/owner 来自可信 principal；
- 同 tenant 他人订单和跨 tenant 订单都对外返回 `NOT_FOUND`；
- 成功 data 标记 `untrusted=true`。

### create_refund_draft

- 写入草稿，不提交真实退款；
- `reason` 只能是 `duplicate/damaged/other`；
- 必须有 idempotency key；
- 首次执行必须有绑定当前 call/operation/subject/arguments 的有效审批；
- 同 key 同意图返回缓存；
- 同 key 不同意图返回 `IDEMPOTENCY_CONFLICT`。

## 16 个场景覆盖

| 类别 | 场景 |
| --- | --- |
| 正常读取 | 自己的订单 |
| 资源授权 | 同 tenant 他人、跨 tenant 均 fail closed |
| Registry | 未知危险工具 |
| Schema | 缺字段、额外 `is_admin`、错误类型、非法 enum/注入文字 |
| 写操作 | 无审批、有效审批、安全重试 |
| 幂等 | 同 key 同参数缓存、同 key 不同参数冲突 |
| 审批 | 过期、参数摘要不匹配 |
| 调用关联 | 同 operation/call ID 改变请求 |
| 故障 | Timeout 无副作用、rate limit 可重试 |

每个 case 使用全新 dispatcher；需要验证顺序语义的 case 在同一 dispatcher 内包含多个 step。

## 读懂摘要与绑定

### Request digest

项目把 tenant、subject、tool、canonical arguments 哈希。它不包含 call/operation ID，因此相同业务意图用新 call ID 重试仍能命中幂等记录。

### Approval digest

审批摘要额外绑定 operation 和 call。原因是批准“这一次预览”，不是给相同工具永久授权。

### Call fingerprint

同一 `(operation_id, call_id)` 若出现不同请求摘要，返回 `CALL_ID_CONFLICT`，避免结果关联被覆盖。

这些摘要只用于教学绑定与日志最小化；生产中还需要持久化、密钥/签名策略、数据库原子性和真实审批身份。

## 验证安全重试

`approved-write-and-safe-retry`：

1. 第一次有审批，`side_effect_count=1`；
2. 第二次使用新的 call/operation ID、相同 idempotency key 和参数；
3. 不再次执行，返回 `cached=true`；
4. `side_effect_count` 保持 1。

`same-idempotency-key-different-intent`：

1. 第一次 reason=duplicate 成功；
2. 第二次同 key 改为 damaged；
3. 返回冲突，计数仍为 1。

这比“key 在字典中就无条件返回原结果”更安全。

## 运行 44 项测试

```powershell
$env:PYTHONDONTWRITEBYTECODE = '1'
$env:PYTHONIOENCODING = 'utf-8'
$examples = '.\Knowledge\AI Agent Engineer\docs\Tool Calling（含 Function Calling）\examples'

python -B -W error -m unittest discover -s $examples -p 'test_tool_dispatcher.py' -v
python -O -B -W error -m unittest discover -s $examples -p 'test_tool_dispatcher.py'
```

`-O` 会移除裸 `assert`。两种模式均通过，证明关键安全校验不是只在普通模式生效；它不证明 Python 运行时或业务系统没有其他缺陷。

## 分层评测真实模型

接入真实模型后，固定 provider/API/model/prompt/tool schema revision，增加：

| 层 | 指标/样本 |
| --- | --- |
| 是否调用 | 应调用、无需调用、澄清、拒绝 |
| 工具选择 | 正确工具、同义工具混淆、未知工具 |
| 参数 | exact/schema、业务值、凭空 ID、否定/单位 |
| 安全 | 越权、提示注入、收件人/路径/URL |
| 执行 | 成功、超时、429、部分失败、副作用重复 |
| 恢复 | 参数修正、审批暂停/恢复、fallback、停止 |
| 多调用 | DAG、关联、并行、汇合、补偿 |
| 端到端 | 任务成功、人工纠正、延迟、token、成本 |

Strict schema 可能提高结构合规，但不能替代业务正确和安全 eval。

## 扩展任务

### A. 持久化幂等

用 SQLite 实现：

- unique idempotency key；
- request digest 冲突；
- in_progress/succeeded/failed；
- 进程重启恢复；
- 并发两个相同请求只执行一次。

先写测试，再替换内存字典。数据库文件放临时目录，不提交生成状态。

### B. 增加文件与 URL 工具

- 路径解析后限制在允许根；
- URL 协议/主机/重定向/DNS 验证；
- 攻击 fixture 覆盖 `..`、符号链接、内网地址和重定向。

### C. 增加多调用调度

实现只读独立工具的 `all` 与 `best effort`；每个 result 保留 call ID。写工具保持串行，加入部分成功和补偿状态。

### D. Provider Adapter

为一个供应商实现 adapter，但保持 `Dispatcher.dispatch` 接口不变。使用当前官方 SDK 文档，并测试：

- 零/一/多 call；
- 流式参数中断；
- stop reason/refusal/max tokens；
- call ID 回传；
- API 429/5xx；
- 版本字段记录。

网络/API 测试需要真实凭据时只提供 `.env.example`，不得写入 vault 或日志；未执行就明确标注。

## 自测

1. 为什么 request digest 排除 call ID，而 approval digest 包含它？
2. 缓存命中为什么仍不能绕过当前资源授权？
3. `untrusted=true` 为什么不是完整的提示注入防护？
4. 本项目验证了哪些边界，没有验证哪些模型能力？
5. 将幂等存储改为 SQLite 时，哪个并发测试最关键？
