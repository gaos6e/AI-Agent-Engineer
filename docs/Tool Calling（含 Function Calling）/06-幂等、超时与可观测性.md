# 幂等、超时与可观测性

## 本节目标

- 防止网络重试、进程恢复和模型重复调用产生重复副作用；
- 正确处理“下游可能成功但响应丢失”的不确定状态；
- 区分 call、operation、approval 与 idempotency 标识；
- 建立可审计 trace、指标和故障恢复。

## 幂等的直觉

幂等操作重复执行一次或多次，预期业务效果与执行一次相同。HTTP RFC 9110 对方法语义定义了 idempotent 概念，但业务工具不能只看 HTTP method：

- `GET order` 通常天然幂等；
- `POST create_refund` 可能重复创建；
- `send_email` 即使请求相同也会重复发送；
- `set_status("closed")` 可能幂等，但触发的通知未必幂等。

必须按业务效果设计。

## 稳定幂等键与请求摘要

写操作接收客户端/编排层产生的 idempotency key。服务持久化：

```text
idempotency_key
tenant / subject / action
canonical_argument_digest
status: in_progress | succeeded | failed
result reference
created_at / expires_at
```

规则：

- 同 key + 同请求摘要：返回原状态/结果，不再执行；
- 同 key + 不同摘要：`IDEMPOTENCY_CONFLICT`；
- key 不存在：原子创建 in_progress，再执行；
- 结果过大：保存引用，不复制整份敏感数据；
- 缓存/记录需跨进程持久化，内存只适合教学。

摘要示例：

$$
d_{\text{request}} = H(
tenant,\ subject,\ tool,\ canonical(arguments)
)
$$

Canonical JSON 要固定键顺序、Unicode 和数字表示。摘要不应包含临时 call ID，否则同意图的模型重试无法命中。

## 为什么“正好一次”很难

场景：

1. 下游成功创建退款；
2. Dispatcher 在保存结果前崩溃；
3. 客户端只看到超时；
4. 重试可能再次创建。

常用策略：

- 下游本身接受同一幂等键；
- 在同一事务中写业务状态与 outbox；
- 先持久化 intent/in_progress，再执行；
- 恢复时查询下游操作状态；
- 用唯一约束防重复；
- 对不可判定状态转人工，不盲目重试。

不能仅凭“请求超时”判断副作用未发生。

## Call ID、Operation ID 与 Key

| 标识 | 是否可跨模型重试稳定 | 主要用途 |
| --- | --- | --- |
| call ID | 通常不保证 | 关联一次 call/result |
| operation ID | 应稳定 | 追踪整个任务/工作流 |
| idempotency key | 对同一业务意图稳定 | 抑制重复副作用 |
| approval ID/digest | 绑定确切动作 | 人类授权与审计 |
| downstream request ID | 下游生成 | 与依赖排障 |

不要把一个 ID 承担所有职责。

## Deadline 与重试

预算层级：

- 连接、读取与单 attempt timeout；
- 单工具 deadline；
- 单模型回合 deadline；
- 整个 operation deadline；
- 审批等待 expiry。

重试规则：

- 只重试确定为 transient 的错误；
- 使用指数退避与 jitter；
- 遵守 `Retry-After`；
- 检查剩余总 deadline；
- 写操作先查幂等状态；
- 限制 attempts 与并发；
- 熔断持续故障依赖；
- Retry budget 本身纳入指标。

参数错误、无权限、审批无效和幂等冲突默认不自动重试。

## 审批与幂等顺序

首次写入：

```text
schema/auth → idempotency conflict check → approval binding → execute → persist result
```

已成功的同 key/同摘要重试可在重新确认当前主体仍有权读取结果后返回缓存，而不再次产生副作用。若权限已撤销，不能因为缓存存在就泄露数据。

## 可观测性

### Trace 字段

- operation ID、provider response ID、call ID；
- tool/schema/adapter/handler revision；
- tenant/subject 的脱敏标识；
- idempotency key 哈希、request digest；
- approval ID/policy revision；
- start/end、queue、attempt、deadline；
- result/error code、cached、retryable；
- downstream request ID；
- 参数摘要与数据分类；
- 模型回合、token/成本与终止原因。

不要记录密钥、认证头、完整敏感参数或未经必要性评估的 tool result。

### 指标

- 工具选择/参数合规；
- auth denied、approval required/expired；
- idempotency hit/conflict；
- side-effect duplicate（应为 0）；
- timeout、rate limit、retry exhausted；
- p50/p95/p99；
- partial success/compensation；
- per-tool 成功、成本和任务完成。

平均延迟会掩盖尾部。按工具、tenant、风险和版本切片。

## 动手实践

设计 `create_refund` 的持久化表和恢复流程，覆盖：

1. 第一次成功；
2. 同 key 同参数重试；
3. 同 key 不同 amount；
4. 下游成功、结果落库前崩溃；
5. 审批过期；
6. 用户权限在重试前撤销；
7. 下游返回 429 和 `Retry-After`。

为每步写出状态转移、是否可重试和用户可见状态。

## 常见错误

- 内存字典当生产幂等存储；
- 同 key 不比较参数摘要；
- 摘要包含临时 call ID；
- Timeout 后直接再发写请求；
- 只监控成功率，不监控重复抑制/冲突；
- 缓存命中跳过当前授权检查；
- 日志记录完整 tool arguments 和凭据。
