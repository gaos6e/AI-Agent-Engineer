# 内容质量与来源标记规范

本规范让学习者和维护者能区分“谁写的、依据什么、核验到何时、能否公开”，避免把原创讲解、资料整理和第三方复刻混成同一种内容。它适用于新建页面和完成深审的既有页面；尚未带这些字段的历史页面视为**未分类**，不能据字段缺失推断为原创或已验证。

## 来源类型

`content_origin` 使用以下有限值：

| 值 | 含义 | 最低要求 |
| --- | --- | --- |
| `original` | 论证结构、讲解、练习或示例由本项目独立创作；外部来源只用于支撑事实和边界 | 列出关键一手来源；动态事实记录 `source_checked`；示例写清实际验证范围 |
| `curated` | 围绕一个或多个明确来源重新整理、摘要或改写，内容组织明显依赖这些材料，但不是近似复刻 | 标出主要来源、获取日期和改写边界；不能把来源观点写成本项目独立结论 |
| `third-party` | 保留第三方原文、近似翻译、镜像、附件或代码 | 记录上游 URL/commit、作者或版权方、许可证及本地改动；许可不明时只在私有层保存，公开层生成来源 stub |
| `mixed` | 同一页面确实包含无法合理拆分的多种来源类型 | 在章节旁逐段标明来源类型；能拆分时优先拆成独立页面 |

“原创”不表示事实无需引用，“整理”也不表示可以省略版权和许可。仅更换措辞不能把第三方复刻变成原创。

## 质量状态

`content_status` 描述证据状态，不描述文章是否“看起来完整”：

| 值 | 含义 | 使用规则 |
| --- | --- | --- |
| `validated` | 已按当前页面范围核对关键事实、链接和相邻实践 | 仍需保留核验日期；不能外推到未测试环境 |
| `dynamic` | 模型、SDK、API、协议扩展、价格、法规或产品行为会快速变化 | 页面必须记录具体日期/版本，并把稳定原理与产品快照分开 |
| `needs-review` | 已知存在来源、公式、实现或时效性缺口 | 不进入主线掌握标准；醒目标出不能依赖的部分 |
| `frozen-reference` | 为历史或上游对照保留，不再作为当前推荐实践更新 | 主线必须有替代入口；保留上游版本、许可和冻结原因 |

`content_tier`（核心、进阶、前沿、实践、参考）回答“怎样学习”，不能代替 `content_origin` 或 `content_status`。

混合型课程入口可以额外使用 `reference_layer_status` 和 `reference_layer_license` 描述其下属参考层，例如“入口页持续维护，但固定译文层冻结且许可未知”。这两个字段只描述入口明确点名的子层，不能替代各第三方正文自己的 `content_origin`、`content_status`、`source_url` 和 `license`，也不能被当作整个入口页的统一许可证。

## 推荐 Frontmatter

```yaml
---
title: 示例页面
content_origin: original
content_status: dynamic
source_checked: 2026-07-19
source_baseline: 官方规范、当前 SDK 文档与本地测试
execution_verified: 2026-07-19
---
```

第三方材料按需追加：

```yaml
source_url: https://example.org/upstream
source_commit: 0123456789abcdef
retrieved: 2026-07-19
license: Apache-2.0
```

不要猜测许可证。缺少再分发证据时使用 `license: unknown` 或正文说明，并保持公开层 stub/exclude 策略。

上述治理字段只使用 frontmatter 根级、块式、单行标量的 `key: value` 写法。不要用 flow mapping、缩进或序列内映射、YAML merge、alias key、治理键上的 tag/anchor、转义键或折叠显式键表达同名字段；公开构建会按 YAML AST 的语义失败关闭，而不是把这些写法当成字段缺失。第三方 `source_url` 只接受不含内嵌凭据的绝对 HTTP(S) URL；公开正文还必须匹配已核验的来源/项目注册、该项目允许的许可证与本地许可声明，未登记来源一律生成 stub。

## 页面内的证据边界

每个完成深审的课程至少说明：

1. 哪些是长期工程不变量，哪些是指定日期的产品或版本快照；
2. 哪些结论来自官方规范、原始论文、本地测试或工程推断；
3. 示例实际运行了什么，未运行什么，是否联网、需要密钥或产生费用；
4. 图片、表格和 Mermaid 是原创、重绘还是第三方资产，以及怎样再生成；
5. 何时不应采用本节方案，以及失败后怎样验证和恢复。

引用第三方原文时优先短引并直接链接；大段翻译或完整复刻必须进入 `third-party` 层并记录许可。由多份资料综合出的建议要明确写成“本项目整理/建议”，不能伪装成某个来源的原话。

## 深审与状态更新

一次页面深审按以下顺序进行：

1. 先读当前页面、相邻索引、示例、测试和既有来源；
2. 只用当前代码、官方文档/规范、原始论文等证据核对关键事实；
3. 逐项修复具体错误、缺口和矛盾，不做全库机械替换；
4. 运行最小相关测试、链接检查和必要的渲染验证；
5. 最后才更新 `source_checked`、`content_status` 与执行记录，并在维护记录中说明未覆盖范围。

如果只改了措辞或只抽查部分来源，不应把整个页面升级为 `validated`。

## 当前迁移策略

- 新建页面必须填写 `content_origin` 与 `content_status`。
- 大幅改写或逐章深审的页面在同一批次补齐字段。
- 历史参考层按课程逐项判断，不批量回填或推断许可。
- 缺少字段的页面保持“未分类”状态，并进入后续审阅队列。
- 公开发布仍以 [[All of AI|课程地图]]、项目发布脚本和 `.website/PUBLISHING.md` 的保守规则为最终门禁。
