## Agent 核心深审

### 修复的运行时缺口

离线 `bounded_agent.py` 新增或收紧以下确定性边界：

- proposal 在访问字段或集合运算前先验证 `ActionProposal`、字符串、参数对象和幂等键类型，畸形 action 进入 `policy_violation`，不再抛出未分类 `TypeError`；
- approval 的 action ID、fingerprint、state version、decision 与 expiry 先做严格类型/非负整数检查，畸形 expiry 失败关闭到 `invalid_approval`；
- lookup tool result 必须具有精确字段、当前 ticket 和预期基础类型，错目标或畸形结果进入 `failed / invalid_tool_result`，不会被记录为可继续使用的 observation；
- 合法查询若确认 ticket 已关闭，runtime 以 `already_satisfied` 完成，不提出 close、不请求写审批，也不制造多余副作用；
- terminal transition 清除可执行 `pending_action`，避免 cancel、failure 或 budget exhaustion 后旧审批仍挂在活动状态；
- checkpoint 恢复验证 observation 内层 hash、evidence 结构、event sequence/state version 连续性、completed action ID 类型与唯一性；
- `waiting_approval` 必须同时有冻结 action、前置 lookup action 与 observation；非等待态不得保留 pending action；
- `completed` 必须精确匹配一条证据路径：`already_satisfied` 只有 lookup 动作与已关闭 observation，且没有 close/evidence；写入完成则必须有 lookup/close 两个动作，以及绑定当前 action fingerprint、ticket、`closed` 状态和非空 receipt ID 的 evidence。

这些检查阻止的是程序错误、迁移错误和不可能状态。普通 SHA-256 仍只能发现偶然损坏；能控制存储的攻击者可以伪造相互一致的 payload/hash/evidence，因此课程继续明确要求受保护存储、MAC/签名、访问控制和外部 receipt 核验。

## 上下文工程深审

### 去重安全边界

原选择器会按 `dedupe_key` 在优先级、日期、大小和 ID 之间选一个候选，但没有验证同组成员是否属于相同 section、权限与信任类别。配置错误时，一个高优先级的 `user-input` 或低权限片段可能替换本应独立处理的受控来源。

`context_budget.py` 现在在选择前拒绝任何跨越以下边界的去重组：

- `section`；
- `required_permission`；
- `trust`。

原有“required chunk 不能与其他成员共享 key”规则继续保留。新增三项负向测试分别证明跨 section、跨权限和跨信任组都会在 fixture 加载时失败；Context Pack 测试由 15 项增至 18 项。

### 可信控制面说明

[[上下文工程/03-选择、相关性与来源|选择、相关性与来源]] 与 [[上下文工程/08-Context Pack项目与自测|Context Pack 项目]] 现在显式区分：

- 身份与 `granted_permissions` 来自认证/授权上下文；
- `allowed_trust` 来自应用策略；
- chunk 的 `trust`、`required_permission`、`required` 和 `dedupe_key` 由受控摄取流程或人工审核赋值；
- 网页、用户输入和工具正文不能自行声明信任、授权、必选或等价关系。

教学 fixture 为离线可读性把 request 与 chunks 放在同一文件；选择器只能检查声明的一致性，不能替这些声明完成身份鉴别。`estimated_tokens` 仍是外部教学估算，不能冒充目标模型 tokenizer、usage 或费用证据。
