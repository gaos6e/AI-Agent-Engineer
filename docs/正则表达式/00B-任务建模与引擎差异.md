# 正则表达式：任务建模与引擎差异

## 本节目标

在学习符号前，先回答三个问题：任务是否需要正则、要调用哪种匹配 API、实际运行的是哪个引擎。这样可以避免“在线测试器能匹配，程序里却失败”的常见问题。

## 先选择合适的工具

正则是文本模式工具，不是所有字符串任务的默认答案。

| 任务 | 优先工具 | 原因 |
| --- | --- | --- |
| 判断固定词是否出现 | 字符串 `in`、`Contains` 或字面搜索 | 规则最简单，也没有模式注入。 |
| 文件名 `*.log` 过滤 | shell 通配符（glob） | `*` 的语义比正则更符合文件枚举。 |
| 提取固定格式日志字段 | 正则或结构化日志解析器 | 输入契约简单时正则合适；JSON Lines 优先用 JSON 解析器。 |
| 解析任意 JSON、HTML 或代码 | 专用解析器 | 嵌套、转义和错误恢复超出普通正则的可靠边界。 |
| 判断自然语言意图 | 分类器、规则组合或 LLM | 文本含义不是字符形状。 |

> [!tip] 最小决策规则
> 能用清晰的字面操作完成，就不用正则；数据有正式语法或嵌套结构，就用解析器；只有当目标确实是“字符模式”时再写正则。

## 四类任务决定 API

同一个模式配上不同 API，会得到不同结果。写模式前先给任务命名：

1. **搜索（search）**：某处是否存在命中，例如日志行是否含 `level=ERROR`。
2. **完整校验（full validation）**：整个输入是否符合契约，例如 run ID 只能由 3～32 个 ASCII 字母、数字、下划线和连字符组成。
3. **提取（extract）**：找到全部命中并读取字段、文本和位置。
4. **替换（replace）**：在已确认范围内重写匹配片段。

以 Python 为例：

```python
import re

text = "ok run_id=r1; retry run_id=r2"
pattern = re.compile(r"run_id=(?P<run_id>[A-Za-z0-9_-]+)")

assert pattern.search(text) is not None
assert [match.group("run_id") for match in pattern.finditer(text)] == ["r1", "r2"]
assert re.fullmatch(r"[A-Za-z0-9_-]{3,32}", "agent_01") is not None
```

`search` 允许前后存在其他文本；`fullmatch` 要求整个字符串符合模式；`finditer` 逐个返回带位置和捕获组的 `Match` 对象。不能用一次成功的局部搜索证明完整输入有效。

## 常见引擎不是同一种语言

| 环境 | 常用入口 | 关键差异 |
| --- | --- | --- |
| Python `re` | `search`、`fullmatch`、`finditer`、`sub` | `str` 模式默认采用 Unicode；命名组为 `(?P<name>...)`；后行断言要求固定长度；3.11 起支持原子组和占有量词。 |
| PowerShell / .NET | `-match`、`-replace`、`Select-String`、`[regex]` | 使用 .NET 引擎；比较操作符默认忽略大小写；命名组为 `(?<name>...)`；替换引用使用 `$1` 或 `${name}`。 |
| ECMAScript | `/pattern/flags`、`RegExp` | `g` 控制继续查找；命名组为 `(?<name>...)`；特性支持取决于 JavaScript 运行时版本。 |
| GNU grep | BRE、`grep -E`、`grep -P` | 默认是 Basic Regular Expression，`-E` 是 Extended；不要把 Python 语法直接复制过去。`-P` 是否可用及细节取决于构建。 |
| Google RE2 | 使用该库的产品或 SDK | 保证匹配时间随输入长度线性增长，但不支持反向引用和 lookaround 等需要回溯的构造。 |

这里的“支持”只说明语法能力，不说明某种引擎适合所有任务。部署前记录语言版本、运行时、调用 API 和模式标志，才能复现结果。

## Unicode 与 ASCII 必须明确

`\d`、`\w` 和 `\b` 看似通用，实际范围由引擎和标志决定：

- Python 的 Unicode `str` 模式中，`\d` 可匹配所有 Unicode 十进制数字，`\w` 包含 Unicode 字母数字和下划线。
- Python 加 `re.ASCII` 后，`\d`、`\w`、`\b` 等收窄为 ASCII 语义。
- ECMAScript 的 `\d` 是 ASCII 数字；.NET 的默认 `\d` 按 Unicode 十进制数字类别解释。
- `\b` 依赖 `\w` 与非 `\w` 的边界，所以它不等同于自然语言“词边界”。

协议字段、端口、时间戳或机器标识若只允许 ASCII，直接写 `[0-9]`、`[A-Za-z0-9_-]` 往往比依赖默认字符类更清楚。自然语言文本则应先写明 Unicode 需求，不要机械改成 ASCII。

```python
import re

assert re.fullmatch(r"\d+", "１２３") is not None
assert re.fullmatch(r"[0-9]+", "１２３") is None
assert re.fullmatch(r"\d+", "１２３", flags=re.ASCII) is None
```

上例中的字符是全角数字。它展示“形状相似”与“契约允许”是两回事。

## 两层转义

模式经常先由宿主语言解析，再交给正则引擎：

```python
import re

raw_pattern = re.compile(r"file\.txt")
ordinary_pattern = re.compile("file\\.txt")
assert raw_pattern.pattern == ordinary_pattern.pattern
```

Python raw string 减少的是 **Python 字符串层** 的反斜杠处理；`\d` 仍由 **正则层** 解释为字符类。PowerShell 中模式优先放在单引号字符串，避免 `$` 被当作变量展开：

```powershell
$pattern = '^run_id=(?<runId>[A-Za-z0-9_-]+)$'
if ('run_id=r1' -cmatch $pattern) {
    $Matches.runId
}
```

`-cmatch` 明确区分大小写；普通 `-match` 默认不区分大小写。成功的标量 `-match` 会更新 `$Matches`，不要在很久以后仍假设它保存的是某次旧匹配。

## 从需求到模式的工程流程

1. 写输入契约：编码、允许字符、长度、是否跨行、最大输入大小。
2. 写出正例、反例、边界例和近似失败例。
3. 记录目标引擎、版本、API 与标志。
4. 从最小模式开始，每次只加入一个约束。
5. 检查整体匹配、捕获组、位置和替换结果。
6. 对不可信输入限制长度，并检查回溯风险。
7. 一旦出现嵌套语法、复杂转义或难以解释的分支，改用解析器或分步逻辑。

## 练习

1. 为“在日志中查找固定字符串 `ERROR`”选择字符串搜索还是正则，并说明原因。
2. 为“验证整个 run ID”写 3 个正例、3 个反例和最大长度边界，不急着写模式。
3. 分别在 Python 与 PowerShell 运行本页示例，记录引擎、API、是否区分大小写。
4. 判断 JSON 对象、Markdown 标题和简单版本号三类输入中，哪些可由小正则辅助，哪些应交给解析器。

## 自测

- 为什么“模式相同”不能保证 Python、PowerShell 与 grep 的结果相同？
- `search` 成功为何不能证明整个输入有效？
- Python raw string 解决的是哪一层转义？
- RE2 为什么会放弃反向引用和 lookaround？
- 机器协议只允许 ASCII 数字时，为何 `[0-9]` 比默认 `\d` 更明确？
