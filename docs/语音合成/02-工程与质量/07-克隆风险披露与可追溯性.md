# 克隆风险、披露与可追溯性

## 本节目标

识别冒充、欺诈和未经授权克隆风险，设计声音许可、使用披露、来源记录、检测边界和事件响应。

## 风险不只来自模型

声音克隆可能被用于冒充亲友、领导、客服或公众人物，也可能绕过依赖声音的人工判断。风险链包括参考音频获取、模型/声音创建、脚本生成、分发渠道和受害者处置。仅在音频末尾加一句免责声明无法覆盖实时诈骗或被剪掉的片段。

## 分层控制

1. **创建前**：验证声音权利、用途、申请者和高风险限制。
2. **生成时**：内容政策、速率限制、敏感场景拦截、审计 ID。
3. **输出时**：适用时做清晰披露，绑定来源、声音和配置记录。
4. **分发时**：渠道验证、异常监控和用户举报。
5. **事件后**：暂停声音、保全审计记录、通知相关方、支持删除与纠正。

水印或来源凭证可作为一层证据，但可能在压缩、重录、裁剪或格式转换后丢失；检测器也有误报和漏报。不能把“未检测到”解释为“真实”，也不能把元数据当作唯一信任根。

## 最小可追溯记录

记录 `request_id`、输入摘要、声音 ID/版本、授权引用、用途、模型与配置、生成时间、操作者/服务身份、输出摘要、披露策略和后续撤销状态。敏感原文不应为了审计无限期保留；摘要和受控证据要符合保留政策。

## 练习与自测

- 为“公司培训旁白”和“模拟 CEO 语音”分别做风险分级；后者即使内部使用也需严格授权与防外泄。
- 设计一个用户报告疑似冒充音频后的 6 步响应流程。
- 解释为什么声音相似度高不能证明获得授权。
