# 项目：可恢复任务工作流

## 项目目标

用 Python 标准库实现并验证一个离线工作流：规则路由、只读检查并行扇出与汇聚、高风险审批、严格 JSON 检查点、错误分类、动作指纹、幂等回执和“动作提交后崩溃”恢复。

脚本是 `00-初学者路线/examples/resilient_workflow.py`，测试是 `00-初学者路线/examples/test_resilient_workflow.py`。它不调用模型、网络或真实业务系统，不需要密钥。项目演示的是架构语义，不是生产审批、分布式锁或持久化方案。

## 你将看到的状态机

```text
start
  ├─ low  -> checks -> evaluate -> execute -> done
  └─ high -> awaiting_approval
                 ├─ reject  -> canceled
                 └─ approve -> checks -> evaluate -> execute -> done

checks 失败 -> failed
execute 提交后崩溃 -> 仍停在 execute -> 根据 receipt 恢复 -> done
```

两个检查 worker 不修改共享状态；主线程 join 后一次写入结果。高风险审批绑定规范化动作的 SHA-256 指纹。动作通过稳定 `action_id` 写本地回执，恢复时若回执一致就复用，不重复提交。

> [!warning] 教学边界
> 检查点哈希不含密钥，只能检测意外损坏或普通篡改，不提供身份认证。原子替换也不等于数据库事务。生产系统仍需并发控制、访问控制、加密、审计保留、迁移和灾备。

## 环境准备

在 `Agentic Design Patterns` 目录打开 PowerShell 7。使用当前稳定 Python 3；本项目无第三方依赖，不需要虚拟环境。创建一次性临时目录：

```powershell
$demo = Join-Path $env:TEMP ("agentic-pattern-" + [guid]::NewGuid())
New-Item -ItemType Directory -Path $demo | Out-Null
$script = Resolve-Path .\00-初学者路线\examples\resilient_workflow.py
```

`$demo` 位于当前用户临时目录，不会向 vault 写运行产物。

## 实验一：低风险直接完成

```powershell
python -B $script `
  --checkpoint "$demo\low-state.json" `
  --receipt "$demo\low-receipt.json" `
  --task-id "low-001" `
  --risk low
```

预期终态为 `done`，事件包括路由、并行汇聚、验收通过与动作提交。打开两个 JSON 文件，检查状态与动作回执是否使用相同 `task_id` 和动作指纹。

## 实验二：高风险暂停、批准与拒绝

第一次运行不提供决定：

```powershell
python -B $script `
  --checkpoint "$demo\high-state.json" `
  --receipt "$demo\high-receipt.json" `
  --task-id "high-001" `
  --risk high
$LASTEXITCODE
```

预期终态 `awaiting_approval`，退出码为 `3`，且回执文件不存在。使用同一 task、risk 和检查点恢复：

```powershell
python -B $script `
  --checkpoint "$demo\high-state.json" `
  --receipt "$demo\high-receipt.json" `
  --task-id "high-001" `
  --risk high `
  --decision approve
```

预期终态 `done`。另起一组新路径，把 `approve` 改为 `reject`，预期终态 `canceled`、退出码 `4`、无回执。拒绝不会被当成可重试错误。

## 实验三：分类重试

模拟 policy 检查第一次出现瞬时超时：

```powershell
python -B $script `
  --checkpoint "$demo\retry-state.json" `
  --receipt "$demo\retry-receipt.json" `
  --task-id "retry-001" `
  --risk low `
  --simulate-transient-once
```

预期完成，`attempts.policy` 为 `2`。再用新路径加 `--simulate-permanent-policy-failure`，预期 `failed`、退出码 `1`，policy 只执行一次且不创建回执。这证明“是否重试”取决于类别，不取决于失败文本长短。

## 实验四：动作已提交但检查点未更新

```powershell
python -B $script `
  --checkpoint "$demo\crash-state.json" `
  --receipt "$demo\crash-receipt.json" `
  --task-id "crash-001" `
  --risk low `
  --crash-after-commit
$LASTEXITCODE
```

预期退出码 `5`。此时 receipt 已存在，而 checkpoint 仍停在 `execute`。去掉模拟开关，使用完全相同参数再次运行：

```powershell
python -B $script `
  --checkpoint "$demo\crash-state.json" `
  --receipt "$demo\crash-receipt.json" `
  --task-id "crash-001" `
  --risk low
```

预期事件为 `action:recovered_existing_receipt`，不是第二次提交。若把另一个任务的 receipt 放到该路径，程序会拒绝覆盖。

## 退出码契约

| 退出码 | 含义 |
| --- | --- |
| `0` | `done` |
| `1` | 确定性检查失败 |
| `2` | 输入、检查点或回执不安全 |
| `3` | 等待审批，稍后可恢复 |
| `4` | 人工拒绝，任务已取消 |
| `5` | 教学用“提交后崩溃” |

自动化调用方应根据退出码和结构化 JSON 共同判断，不解析自然语言日志。

## 阅读代码的顺序

1. `new_state` / `validate_state`：严格 schema、版本和不变量。
2. `save_state` / `load_state`：封装哈希与原子替换。
3. `route`：规则优先的风险路由。
4. `run_parallel_checks`：worker 无共享写入，主线程统一 join。
5. `perform_action`：稳定 action ID 与回执恢复。
6. `run`：显式状态转换、审批和终态。
7. `main`：CLI 参数、JSON 输出和退出码。

## 运行测试

在 examples 目录执行：

```powershell
Push-Location .\00-初学者路线\examples
$env:PYTHONDONTWRITEBYTECODE = "1"
python -B -m unittest -v test_resilient_workflow.py
python -B -O -m unittest -v test_resilient_workflow.py
python -B -W error -m unittest test_resilient_workflow.py
Pop-Location
```

测试覆盖严格状态校验、检查点篡改、路由、并行汇聚、瞬时与永久错误、暂停稳定性、批准、拒绝、完成后重复恢复、提交后崩溃和 CLI 退出码。`unittest` 断言在 `-O` 下仍有效。

## 综合任务

在不降低现有测试的前提下，增加第三个只读分支 `compliance`：

1. 更新分支常量、默认检查和严格 schema；
2. 让汇聚器拒绝缺少该分支；
3. 增加 compliance 瞬时失败、永久拒绝和完整成功测试；
4. 解释它为何可与另外两个分支并行；
5. 记录测试数量与正常、`-O`、warnings-as-errors 三种运行结果。

完成标准不是“代码能启动”，而是新分支缺失时必须失败关闭、已有恢复语义不变。

## 项目验收

- [ ] 低风险路径达到 `done`，回执与任务匹配。
- [ ] 高风险无决定时暂停，拒绝不产生回执。
- [ ] 审批绑定当前动作指纹，动作变化会被拒绝。
- [ ] 只有 `transient` 检查在预算内重试。
- [ ] 检查点损坏、未知字段和版本不符都失败关闭。
- [ ] 提交后崩溃能凭相同回执恢复，不重复动作。
- [ ] 正常、`-O` 与 warnings-as-errors 测试均通过。
- [ ] vault 中没有检查点、回执、缓存、密钥或真实凭据。
