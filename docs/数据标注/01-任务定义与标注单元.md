# 任务定义与标注单元

## 本节目标

把抽象构念转成标注者可观察、可复核的单元、上下文、标签与证据规则，同时阻止部署时不可见信息泄漏进标签定义。

## 先确定标注服务哪个决策

“标注回答质量”过于模糊。应写成：谁使用标签、在哪个时点、依据哪些可见证据、错误的后果是什么。例如：评审者只依据用户问题、允许的工具结果和最终回答，判断回答是否存在会改变用户行动的事实错误。

## 标注单元

标注单元是一次独立判断的最小对象：

- 文本分类：一条完整查询。
- 实体抽取：文本中的一个字符跨度及实体类型。
- RAG 相关性：一个“查询—chunk”对。
- Agent 轨迹：一个工具调用、一个步骤或整条运行。
- 生成回答：回答整体或拆成事实性、完整性、安全性等维度。

单元太大会混合多个问题；太小则失去上下文。必须规定上下文窗口和允许查看的证据，避免不同标注者掌握不同信息。

若标签用于训练部署时的预测器，标注证据不能包含预测时尚未发生的信息。例如用“用户后来退款”定义当时请求是否高风险，可能合理地作为事后结果标签；但模型输入与数据划分必须严格保证该未来结果不会泄漏到特征。

## 标签空间

标签应互斥、尽量穷尽，并允许处理不可判断情况：

```text
helpful：回答正确、相关，足以支持当前任务
not_helpful：相关但存在关键遗漏、错误或不可执行
unsafe：建议会造成明确安全/权限风险
cannot_judge：证据缺失或任务本身无法判断
exclude：样本损坏、重复或不在任务范围
```

`cannot_judge` 是有效判断；`exclude` 表示样本不应进入本任务。不要把二者都偷偷映射为负类。

## 标签层级与证据

复杂任务可拆成多个可观察维度，再由明确规则派生总标签。例如“事实正确、引用支持、完成约束、安全”分别判断；总通过要求四项都通过。这样比主观 1～5 分更容易复核。

每个重要标签最好附证据：引用文本跨度、工具步骤 ID、文档来源或一句理由。证据使裁决与错误分析可行。

## 练习

把“判断 RAG 回答好不好”改写为一张任务卡，包含单元、上下文、标签、排除条件、证据字段和至少两种错误代价。
