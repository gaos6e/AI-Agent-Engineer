# 写入、更新、删除与版本

## 本节目标

你将设计一条可重放、可对账的数据生命周期：稳定 ID 的幂等 upsert，来源/权限/向量一致更新，部分失败恢复，删除 tombstone 和物理清理，以及模型/索引双版本发布。

## 写入来自 canonical revision

一个 point 不应由“某次脚本临时算出的向量”定义，而应绑定：

```text
tenant + point_id
source_id + source_revision
chunk_id/span + actual input hash
embedding space + revision
payload/ACL revision
ingestion job + attempt
```

上游 `[[Knowledge/AI Agent Engineer/docs/知识库构建/00-目录|知识库构建]]`可用 canonical revision + outbox 产生事件；embedding worker 只处理明确 revision，向量库写入后再回报状态。这样可以从事实源重建，不依赖一次不可重放调用。

## 稳定 ID 与幂等 upsert

相同逻辑 point 的重试应使用相同 ID。Upsert 语义通常是：

- ID 不存在：创建；
- ID 存在：替换/更新；
- 完全相同 contract/source/input hash：no-op；
- 老 revision 晚到：拒绝，不能覆盖新值。

请求可携带：

- event/job ID；
- expected source revision；
- content/input hash；
- expected store/row version；
- embedding contract signature。

产品是否提供 compare-and-set、事务或写 ordering 不同。若没有，应用层仍要保存版本并在写前/写后对账，不能把“upsert”误解为自动解决乱序并发。

## 部分成功

批量写入可能：

- 全部成功；
- 部分 point 成功；
- 客户端超时但服务端已完成；
- 某 shard/replica 成功、另一些待恢复；
- 合法记录与坏维度/坏 payload 混在一批。

可靠流程：

1. 每项有稳定 ID 与 input hash；
2. 保存 provider/database request ID；
3. 解析逐项或批次结果；
4. 对成功项做 read-back/状态确认（按风险）；
5. 只重试未知/临时失败项；
6. 永久坏数据隔离；
7. 用 `expected IDs - published IDs` 闭合差集；
8. 记录重试总预算。

不能因一次 HTTP 200 宣称全批完整，也不能因一次 timeout 全量生成新 ID 重写。

## 更新是多字段一致性问题

正文或 chunk 变化时，至少需要同步：

- vector；
- source revision；
- content/input hash；
- document/chunk mapping；
- ACL/status；
- embedding revision；
- indexed/published revision。

最危险状态是“新 vector + 旧 ACL”或“旧 vector + 新正文”。解决方式取决于产品：

### 单数据库事务

pgvector 在 PostgreSQL 表中时，可利用数据库 transaction 把 vector、关系字段和业务状态一起提交。仍要处理 embedding API 在事务外完成的问题，通常先生成派生结果，再用短事务写入。

### Staging + publish

专用向量库或跨系统流水线可写 staging space/version，完成数量、hash、ACL、quality 对账后原子切 published alias/pointer。

### Versioned records

并行保留 `point_id + source_revision`，查询只读 published revision；切换后清理旧版。增加存储，但回滚清楚。

不要让应用实例各自切换配置，使用统一读指针。

## 权限更新优先级

ACL 变更不一定需要重新生成 vector，但必须立即更新所有可读路径：

- 活动 collection 的 payload；
- old/shadow spaces；
- sparse/inverted index；
- query/result/parent cache；
- replicated copies；
- export/backup 恢复后的撤销日志。

如果向量与 payload 不能原子更新，安全基线是先将 status 设为不可检索，再更新 ACL/vector，验证后重新发布。不要在中间态保留过宽权限。

## 删除的三个层次

### 逻辑不可见

查询立即排除，常用 tombstone、status=deleted 或删除 point。记录 deletion event、tenant、source revision 和时间/序号。

### 索引/存储回收

ANN 图、segment、WAL、compaction 可能延后回收。删除 API 成功不等于磁盘空间立刻下降，也不等于旧 segment 不再存在于备份。

### 保留与合规

同步处理：

- canonical source；
- 所有 chunks/points/named vectors；
- 所有 embedding spaces；
- sparse/keyword index；
- caches、exports、debug artifacts；
- snapshots/backups 的保留与到期；
- 审计证明。

是否能立刻从不可变备份删除取决于治理/法规和系统设计；需记录隔离、到期与恢复后重放删除的机制，不能虚假承诺“任何副本已物理擦除”。

## Tombstone 的作用与风险

Tombstone 可阻止旧事件把已删除 point 意外复活，并帮助恢复/迁移后重放删除。它至少包含：

- point/source ID；
- tenant/security domain；
- delete revision/event ID；
- retention/physical purge 状态。

风险：

- tombstone 自身无限增长；
- ID 被另一 tenant 重用；
- 备份恢复缺 tombstone 导致复活；
- 并发 upsert/delete 顺序不明确。

定义 resurrection policy：同一 tenant 是否允许、需要哪个 source revision/审批；跨 tenant ID 复用默认拒绝。

本库玩具项目保留 `point_id + tenant + deleted_at_revision`，同 tenant upsert 可显式 resurrect，跨 tenant 禁止。它是教学策略，不代表生产合规默认。

## Embedding/索引版本迁移

完整流程承接 `[[Knowledge/AI Agent Engineer/docs/Embedding/04-相似度索引与版本迁移|Embedding 空间迁移]]`：

1. 冻结 old space contract/alias；
2. 创建 new collection/index；
3. 从 canonical chunks 重算，不从旧向量猜；
4. 对账 IDs、source/hash、ACL、tombstone、dimension/norm；
5. old-query→old-index、new-query→new-index 独立评测；
6. 影子/灰度；
7. 原子切读；
8. 停止旧写入；
9. 保留回滚窗口；
10. 按删除/保留政策回收旧空间。

Index 参数变化但 embedding 不变也可用类似 blue/green；这样能区分索引构建失败与模型质量变化。

## 同步模式

### Dual write

应用同时写 source DB 与 vector DB。简单但容易一边成功一边失败；需要 outbox/重试/对账。

### Change Data Capture / Outbox

源事务提交业务状态与 outbox event，worker 异步更新向量库。延迟可见，但事件可重放、事实来源清楚。必须处理重复、乱序和 poison event。

### Periodic reconciliation

定期比较 canonical ID/revision/hash 与向量库，发现漏项、陈旧、孤儿和 tombstone 不一致。它不是替代实时同步，而是最后完整性网。

可组合：outbox 实时 + nightly reconciliation。

## 发布对账

每次 batch/revision：

| 维度 | 检查 |
| --- | --- |
| Inventory | expected/published/missing/extra IDs |
| Version | source、embedding、schema、index build revision |
| Integrity | content/input/vector hashes、dimension、finite/norm |
| Security | tenant/ACL/status、越权测试 |
| Lifecycle | deletes/tombstones/purge queue |
| Quality | canary/gold top-k 与关键子组 |
| Operations | error/retry、searchable lag、index ready |

Extra/orphan points 与 missing points 同样危险；不要只比总数，因为“一缺一多”总数仍相同。

## 常见错误与排查

- **重试生成新 ID**：改稳定 ID + event/input hash。
- **旧事件覆盖新正文**：比较 source revision/CAS。
- **向量和 ACL 分两步且中间可搜**：先撤下 status，或用事务/staging。
- **HTTP 200 当全批成功**：逐项/差集对账。
- **删除后又复活**：恢复/乱序 upsert 没有 tombstone/version。
- **只删当前 collection**：旧空间、cache、sparse index 和备份遗漏。
- **原地覆盖模型向量**：新旧空间混合，改双索引。
- **只比 point count**：加 ID set、hash 与 ACL 对账。

## 练习

1. 为 source revision r7/r8 乱序到达写出接受/拒绝规则。
2. 设计 100 项批写中 97 成功、2 临时失败、1 坏维度的恢复状态机。
3. 画“ACL 收紧 + vector 更新不能原子”的安全发布顺序。
4. 从 document 删除映射到 12 chunks、2 spaces、keyword index、2 caches 和 backup retention。
5. 比较 dual write 与 outbox 的失败场景和对账方式。
6. 设计同 tenant resurrection 与跨 tenant ID reuse 的策略。

## 掌握检查

- [ ] Point 写入绑定 canonical/source/input/embedding revision。
- [ ] Upsert 对相同事件幂等，旧 revision 不覆盖新。
- [ ] 部分/未知成功逐项对账，不整批盲重试。
- [ ] Vector、payload、ACL、status 在可见版本中一致。
- [ ] 删除覆盖逻辑不可见、物理回收与保留治理。
- [ ] Tombstone 防止恢复/乱序复活并限制 ID 复用。
- [ ] 模型/索引迁移使用新空间、评测、原子切换和回滚。
- [ ] 定期 reconciliation 检查 ID 集、hash、ACL 和 delete。
