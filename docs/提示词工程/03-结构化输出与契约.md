# 结构化输出与契约

## 本节目标

区分“像 JSON”“可解析 JSON”“符合 schema”和“业务语义正确”，建立从模型输出到业务动作之间的验证层。

## 四层检查

假设期望输出 `{"label":"billing","confidence":0.8}`：

1. **语法**：能否被 JSON 解析器读取？单引号、尾逗号和 Markdown 代码围栏都可能导致失败。
2. **结构**：字段是否存在、类型是否正确、枚举是否合法、是否出现多余字段？JSON Schema 可描述这层。
3. **业务规则**：`confidence` 是否在允许范围；`unknown` 是否必须附带原因；证据引用是否真的来自输入？
4. **授权**：即使前三层通过，是否允许自动退款、发邮件或删除数据？高影响动作仍需独立授权或人工确认。

供应商的 schema 约束能力、支持的关键字与失败表示会变化。以 2026-07-14 的 OpenAI 文档为例，JSON mode 只保证输出是有效 JSON，不保证匹配你的 schema；Structured Outputs 在受支持的模型和功能上提供 schema 遵循保证。即便如此，应用仍须处理拒答、输出未完成和“结构合法但语义错误”的值。其他供应商是否提供同等保证，必须查其当前文档，不能由名称推断。

## 一个最小 schema

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "properties": {
    "label": {"type": "string", "enum": ["billing", "technical", "other"]},
    "reason": {"type": "string", "minLength": 1, "maxLength": 160},
    "evidence": {"type": ["string", "null"], "maxLength": 120}
  },
  "required": ["label", "reason", "evidence"],
  "additionalProperties": false
}
```

提示中解释字段语义，API 层尽可能启用供应商当前支持的 schema 功能，程序端再解析和校验。三处使用同一份契约，避免提示、API 参数和代码各自漂移。本库可执行的完整版本见 [[提示词工程/examples/response.schema.json|response.schema.json]]；离线验证器还执行“`billing`/`technical` 必须给证据”和“证据必须能在输入中找到”的业务规则，这些规则不能仅靠字段类型表达。

## 失败处理

- 解析失败：记录经过脱敏的响应片段、请求 ID 和提示版本；只在策略允许时重试。
- schema 不符：不要“猜着修”；可发起一次明确的修复请求，仍须验证。
- 内容拒绝或安全拦截：作为显式结果处理，不伪装成空成功。
- 输出截断：缩小任务、提高合理的输出上限或改用分步流程；不能直接拼接半个 JSON。
- 语义不可信：降级为人工复核，不把 `confidence` 当校准概率。

## 练习与自测

为“从发票文本抽取币种和总额”写 schema，并回答：总额为字符串还是数字？千位分隔符如何处理？文本没有币种时是 `null`、`unknown` 还是失败？若这些没有明确，schema 还不是完整合同。
