# RAG

## 知识库简介

RAG（Retrieval-Augmented Generation，检索增强生成）是在回答前从外部知识源取回证据，再让生成模型依据证据组织回答的系统方法。它把“模型参数里记住了什么”与“当前可查、可更新、可引用的资料”连接起来，适合内部知识问答、客服说明、研究助手和有来源要求的 Agent。

初学者最容易把 RAG 简化成“向量检索 + LLM”。工程上的 RAG 实际包含两条管线：

- 离线知识管线：来源接入 → 解析 → 清洗 → chunk → 表示 → 索引 → 质量检查；
- 在线回答管线：身份与路由 → 硬过滤 → 召回 → 重排 → 上下文选择 → 生成 → 引用校验 → 监控。

> [!important] 三个不保证
> RAG 不自动保证来源正确、检索完整或回答忠实。检索到文档也不等于文档支持结论；出现引用标记也不等于引用有效；把私有文档放进向量库也不等于权限安全。

## 学习目标

完成本库后，你应能：

- 画出离线索引与在线回答管线，说明每层输入、输出和责任；
- 为 source、parser、chunk、Embedding、索引、检索、reranker、prompt 和模型建立版本链；
- 区分闲聊、知识检索、实时工具、高风险拒绝和人工升级路由；
- 先做 tenant、ACL、状态与有效期硬过滤，再做相关性排序；
- 组合 sparse/dense 召回、融合、重排和可测量的降级路径；
- 在上下文预算内完成 canonical 去重、来源覆盖、相邻段选择和冲突处理；
- 将回答拆成可核查 claim，并验证引用确实来自本次上下文且支持 claim；
- 对无答案、权限不足、证据冲突、依赖故障和实时问题采取不同策略；
- 分别评测检索、上下文、回答、引用、路由、安全、延迟与成本；
- 用 trace 和回归样本定位错误层，而不是看到错答就只调 prompt。

## 核心术语

| 术语 | 初学者解释 | 工程边界 |
| --- | --- | --- |
| source | 原始资料及其身份、版本和权限 | 网页标题不是稳定来源 ID |
| chunk | 可独立检索的证据单元 | 不一定等于回答上下文 |
| candidate | 第一阶段召回的候选 | 还未必进入 prompt |
| evidence | 通过过滤、重排和预算选择的上下文 | 必须保留来源与版本 |
| grounding | 回答中的主张由给定证据支持 | “听起来合理”不算支持 |
| claim | 可单独核查真假的陈述 | 一个长段落通常含多个 claim |
| citation | claim 到 source/span/revision 的映射 | 只有 `[S1]` 格式不够 |
| abstention | 证据不足时不编答案 | 应区分原因与下一步 |
| trace | 各阶段输入、版本、候选、决策与故障记录 | 对外结果不能泄露被过滤资料 |

## 掌握标准

- [ ] 能分清 source、chunk、candidate、selected context、claim 和 citation。
- [ ] 所有硬权限过滤在相关性打分前完成，fallback 也不能绕过。
- [ ] 每个回答可追溯到 source/chunk/revision、各阶段 rank 与版本。
- [ ] 能解释 candidate recall 高但回答仍错的至少三种原因。
- [ ] 能为无答案、冲突、实时状态、无权限和依赖故障定义不同状态。
- [ ] 每个可核查 claim 均有本次上下文中的支持证据。
- [ ] 被过滤文档的 ID、标题和内容不会出现在用户可见 trace。
- [ ] 离线集覆盖常见、长尾、时效、权限、冲突、攻击和无答案切片。
- [ ] 发布门禁同时覆盖质量、安全、p95/p99、成本和 fallback。
- [ ] 能运行项目与测试，并解释它没有验证真实模型质量的原因。
