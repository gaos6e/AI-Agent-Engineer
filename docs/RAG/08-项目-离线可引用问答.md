# 项目：离线可引用问答

## 项目目标

运行一条完全离线、可追踪、可故障注入的教学管线：

```text
稳定 query fixture → 路由 → tenant/ACL/时间过滤 → 字符召回
                 → topic 规则重排 → canonical 去重与预算
                 → 抽取式 claim → citation/revision 校验 → 状态输出
```

项目解决的是“RAG 的每层契约如何被验证”，不是“如何用字符规则做高质量语义搜索”。它故意不使用 Embedding、ANN、LLM、网络或 API key。

## 项目文件

| 文件 | 内容 |
| --- | --- |
| [[RAG/examples/rag-fixture.json\|rag-fixture.json]] | 9 篇版本化文档、7 个 query 及 qrels/forbidden 期望 |
| [[RAG/examples/offline_cited_qa.py\|offline_cited_qa.py]] | 严格 JSON、过滤、召回、重排、上下文、回答、引用与 CLI |
| [[RAG/examples/test_offline_cited_qa.py\|test_offline_cited_qa.py]] | 37 项 schema、权限、故障、grounding 与命令行测试 |

## 环境

- Windows 11 与 PowerShell 7；
- Python 3.11 已验证，代码只依赖标准库；
- 不需要安装依赖或提供密钥；
- 通过 `-B` 和 `PYTHONDONTWRITEBYTECODE` 避免生成 `__pycache__`。

若你需要隔离 Python，可把 venv 建在 vault 外，例如 `$env:TEMP\rag-lab-venv`；不要把 `.venv`、缓存或真实凭据放进知识库。

## 第一步：运行全部案例

从 vault 根目录执行：

```powershell
$env:PYTHONDONTWRITEBYTECODE = '1'
$env:PYTHONIOENCODING = 'utf-8'
$script = '.\Knowledge\AI Agent Engineer\docs\RAG\examples\offline_cited_qa.py'
$fixture = '.\Knowledge\AI Agent Engineer\docs\RAG\examples\rag-fixture.json'

python -B -W error $script --fixture $fixture demo
```

预期业务状态：

| Query ID | 场景 | 状态 |
| --- | --- | --- |
| `Q-refund` | 当前退款规则，旧版已过期 | `answered` |
| `Q-duplicate` | 重复扣款处理 | `answered` |
| `Q-phone-guest` | 普通用户询问内部电话 | `insufficient_evidence`，且不泄露 `S3` |
| `Q-phone-oncall` | 有 oncall 组权限 | `answered` |
| `Q-conflict` | 两个当前住宿标准冲突 | `conflict` |
| `Q-mars` | 语料无答案 | `insufficient_evidence` |
| `Q-order-live` | 实时订单状态 | `tool_required` |

## 第二步：读懂 trace

单独运行：

```powershell
python -B -W error $script --fixture $fixture ask --query-id Q-refund
```

检查：

1. `filter_summary` 只有聚合原因，不含被过滤文档标题或 ID；
2. `retrieved`、`reranked`、`selected` 分开保存；
3. `dropped` 能看到 canonical duplicate 或预算原因；
4. `claims` 的文本来自 fixture 中逐字 statement；
5. `citations` 同时记录 document、fact 和 source revision；
6. `validate_result` 确认引用来自 selected context 并支持 claim。

字符/bigram 分数只在本 query 内排序，不是概率，也不能与真实向量分数比较。

## 第三步：比较权限

```powershell
python -B $script --fixture $fixture ask --query-id Q-phone-guest
python -B $script --fixture $fixture ask --query-id Q-phone-oncall
```

两次 query 文本相同，可信 `subject_groups` 不同。观察普通用户结果中：

- 没有内部电话；
- 没有 `S3`；
- 没有私有标题；
- 只有聚合的 `acl_denied` 计数。

这是“内部诊断能区分原因、对外结果不枚举私有资源”的最小示范。

## 第四步：观察冲突与工具路由

```powershell
python -B $script --fixture $fixture ask --query-id Q-conflict
python -B $script --fixture $fixture ask --query-id Q-order-live
```

`Q-conflict` 保留两个互相矛盾的有效 fact，不取平均、不凭 authority 随意选一个。`Q-order-live` 在检索前短路，因为知识快照不能回答当前订单状态。

## 第五步：模拟依赖故障

```powershell
python -B $script --fixture $fixture ask --query-id Q-refund --failure retrieval_error
python -B $script --fixture $fixture ask --query-id Q-refund --failure reranker_error
python -B $script --fixture $fixture ask --query-id Q-refund --failure generation_error
```

| 故障 | 行为 |
| --- | --- |
| retrieval_error | 停止并返回 `dependency_unavailable` |
| reranker_error | 在相同安全候选集上保留 retrieval 顺序，标记 degraded |
| generation_error | 保留检索 trace，但不输出未经验证 claim |

## 第六步：运行回归测试

```powershell
$env:PYTHONDONTWRITEBYTECODE = '1'
$env:PYTHONIOENCODING = 'utf-8'
$tests = '.\Knowledge\AI Agent Engineer\docs\RAG\examples'

python -B -W error -m unittest discover -s $tests -p 'test_offline_cited_qa.py' -v
python -O -B -W error -m unittest discover -s $tests -p 'test_offline_cited_qa.py'
```

`-O` 会移除 Python 的裸 `assert`。两种模式都通过，说明关键校验没有依赖会被优化掉的断言；它仍不等于证明所有生产风险已覆盖。

## Fixture 契约

### 文档

每篇文档包含：

- stable/canonical ID、title、text；
- tenant、ACL、status、有效期；
- source revision 与教学 authority；
- 一个或多个 fact：topic、逐字 statement、value 与 unit。

`statement` 必须逐字存在于 `text`，让引用支持可确定验证。真实系统通常使用 source span，而不是预先结构化 fact。

### Query

每个 query 包含：

- 文本、tenant、可信 groups、as_of；
- route 与稳定 topic；
- expected status/facts；
- 不得泄露的 forbidden document IDs。

期望字段服务于离线评测，不应直接传给线上回答器。

## 实践任务

### 任务 A：增加“已删除文档”

新增一个 `status=archived` 的高相似文档和 query。要求：

- Fixture schema 仍通过；
- 文档不进入 retrieved；
- 对外 trace 不出现其 ID；
- 新增测试覆盖正常与 `-O`。

### 任务 B：预算压缩

把 `max_context_chars` 调小，观察 `dropped.reason`。回答被拒绝时，判断是 retrieval miss 还是 context budget 导致。

### 任务 C：从规则组件替换为真实组件

按顺序替换：

1. 字符召回 → BM25，并保留旧版对照；
2. 再增加 Embedding/dense 通道与融合；
3. topic rule → reranker；
4. 抽取生成 → LLM 结构化输出；
5. 逐步增加模型 judge，但保留确定性权限和 citation ID 检查。

每次只替换一层，锁定 revision 并运行同一测试集。

## 生产化迁移清单

- [ ] 身份、tenant 与 group 来自可信认证层；
- [ ] Parser/chunk/embedding/index 都有 revision；
- [ ] 删除、过期和权限变更传播到缓存与副本；
- [ ] Sparse/dense 分开观测，融合有 qrels 实验；
- [ ] Reranker 输出验证未知、重复、缺失与非有限值；
- [ ] Token 预算使用真实 tokenizer；
- [ ] Evidence 使用 source/span/revision；
- [ ] LLM 输出有 schema、超时、限流和失败状态；
- [ ] Claim support 有人工校准，关键字段有确定性检查；
- [ ] 外部文档视为不可信数据，工具最小权限；
- [ ] 日志脱敏、限权、有保留期；
- [ ] 离线、影子、小流量与回滚门禁齐全。

## 项目验收

- [ ] 七个 query 的正常状态符合 fixture；
- [ ] 旧版、跨 tenant 与 ACL 文档先过滤后评分；
- [ ] Canonical 镜像不重复占用上下文；
- [ ] 冲突证据不被合成为第三个答案；
- [ ] 实时问题不走知识检索；
- [ ] Claim 和 citation 都能回溯到 selected context；
- [ ] 三类依赖故障有确定、安全、可观测行为；
- [ ] 37 项测试在正常与 `-O` 模式通过；
- [ ] 没有网络、密钥、缓存、模型或大型数据依赖。

## 自测

1. 为什么 fixture 中的 expected 字段不能作为线上回答输入？
2. 为什么本项目能验证 citation provenance，却不能证明真实 LLM 的所有语义忠实性？
3. Ordinary guest 的 trace 为什么只显示聚合过滤原因？
4. Reranker fallback 为什么不能重新扩大候选权限范围？
5. 把字符检索换成向量检索后，哪些契约不应改变？
