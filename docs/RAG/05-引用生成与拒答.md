# 引用、生成与拒答

## 本节目标

- 把回答拆成可独立核查的 claim；
- 区分“引用存在”“引用可访问”“引用支持主张”；
- 用结构化契约限制生成结果并做确定性校验；
- 为证据不足、冲突、无权限和依赖故障设计不同状态。

## 先定义生成契约

生成器不应只返回一段字符串。一个教学用结果可以是：

```json
{
  "status": "answered",
  "answer": "退款审核通过后通常在一至三个工作日原路返回。",
  "claims": [
    {
      "claim_id": "C1",
      "text": "退款审核通过后通常在一至三个工作日原路返回。",
      "citations": [
        {"document_id": "S1", "fact_id": "F-refund-current"}
      ]
    }
  ],
  "citations": [
    {
      "document_id": "S1",
      "fact_id": "F-refund-current",
      "source_revision": "refund-2026-01"
    }
  ]
}
```

生产结构会按业务变化，但至少应表达：

- 回答状态；
- 用户可读答案；
- 可核查 claim；
- claim 与 source/span/revision 的映射；
- 不确定、冲突或拒答原因。

JSON Schema 只能证明结构正确，不能证明语义得到支持。

## Claim 为什么重要

一句话“退款三天到账，而且一定免手续费”包含两个主张。一个引用可能只支持前半句。把长回答拆成 claim 后，才能逐项检查：

1. claim 是否需要证据；
2. 引用是否来自本次 selected context；
3. source 的具体 span 是否支持这个 claim；
4. source 是否当前有效且用户可访问；
5. 多来源是否一致或冲突。

一般建议、价值判断和推断也要标明性质，不能伪装成来源中的事实。

## 引用校验的四层

### 1. 结构与 ID

- 字段完整且类型正确；
- document/chunk/span ID 存在；
- ID 唯一、无未知或重复项；
- source revision 与本次上下文一致。

### 2. Provenance

- 引用必须来自本次 selected context；
- 不能引用召回过但未进入上下文的文档；
- 不能引用被 ACL、tenant 或有效期过滤的资料。

### 3. Entailment / 支持关系

- 引用片段是否真的支持 claim；
- 条件、否定、数字、时间和适用范围是否一致；
- 推断是否超过证据。

本库项目用“claim 必须逐字等于 fact statement”建立可确定验证的下界。真实生成更灵活，需要人工 rubric、规则、NLI 或 LLM judge 辅助，但高风险场景仍需抽样人工核查。

### 4. 用户可核查性

- 用户能否打开来源；
- 链接是否定位到正确页/段；
- 展示的标题和版本是否明确；
- 私有来源是否按权限渲染。

用户无法打开的“神秘 S1”不能提供完整可核查体验。

## 拒答不是一个状态

| 状态 | 含义 | 用户信息 | 不应做 |
| --- | --- | --- | --- |
| insufficient_evidence | 可访问资料不能支持答案 | 说明资料不足、建议补充或人工核验 | 暗示私有资料存在 |
| conflict | 当前有效来源互相冲突 | 展示冲突边界与来源 | 擅自选一个 |
| tool_required | 问的是实时状态或动作 | 转到鉴权工具/确认流程 | 用知识快照冒充实时值 |
| dependency_unavailable | 检索依赖失败 | 说明暂不可检索 | 让模型凭记忆答内部事实 |
| generation_unavailable | 已取证据但生成失败 | 可按策略返回原文或稍后重试 | 输出未校验草稿 |
| policy_denied | 安全/合规规则拒绝 | 给安全替代路径 | 泄露检测细节 |

“无权访问”与“没有资料”在内部诊断上不同；对外信息要防止通过标题、ID、候选数或错误差异枚举私有内容。

## 参数记忆与证据边界

即使提示写“只依据上下文”，模型仍可能使用参数记忆。对内部制度、医学、法律或财务等可追溯场景：

- 要求所有事实 claim 有引用；
- 删除或拒绝无法对齐证据的 claim；
- 把一般建议单独标明，不混入制度事实；
- 对关键数字和否定做确定性检查；
- 设置人工升级或二次核验。

“模型很自信”不是证据。

## 冲突回答示例

如果两个有效来源分别写“每晚 500 元”和“每晚 600 元”，合格结果不是取平均 550，也不是默认选 top-1：

```text
状态：conflict
结论：现有有效资料不一致，暂不能给出唯一标准。
证据 A：每晚 500 元（来源 A，revision ...）
证据 B：每晚 600 元（来源 B，revision ...）
下一步：请制度负责人确认适用范围或修正文档。
```

若权威层级、地区或生效时间已有确定性规则，可以先按规则消解；规则及版本同样要进入 trace。

## 动手实践

运行以下案例并检查 `status`、`claims`、`citations` 与 `selected`：

```powershell
$env:PYTHONDONTWRITEBYTECODE = '1'
$script = '.\Knowledge\AI Agent Engineer\docs\RAG\examples\offline_cited_qa.py'
$fixture = '.\Knowledge\AI Agent Engineer\docs\RAG\examples\rag-fixture.json'

python -B $script --fixture $fixture ask --query-id Q-refund
python -B $script --fixture $fixture ask --query-id Q-conflict
python -B $script --fixture $fixture ask --query-id Q-phone-guest
python -B $script --fixture $fixture ask --query-id Q-order-live
```

然后在测试的副本中做三次篡改：

1. 把 citation 改成未选中的 document ID；
2. 把 claim 改成“保证当天到账”；
3. 把 source_revision 改成 `wrong`。

说明每次应在哪一层被拒绝。

## 常见错误

- 只用正则确认回答里有 `[S1]`；
- 一个段落只有一个引用，却包含多个独立 claim；
- 引用来自 retrieved 而非 selected context；
- 生成器能自行编造 source ID；
- 无答案时悄悄切换成无引用自由回答；
- 拒答内容暴露被过滤文档的标题或 ID。

## 自测

1. 引用 ID 存在为何仍不足以证明回答有据？
2. Claim-level citation 比段落末尾统一引用解决什么问题？
3. 结构化输出校验和语义支持校验有何区别？
4. 为什么实时订单问题应该返回 tool_required？
5. 权限不足时，用户可见结果为何不能包含被过滤 source ID？
