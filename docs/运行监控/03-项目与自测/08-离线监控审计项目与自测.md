# 离线监控审计项目与自测

## 项目目标

在不连接监控平台、云账号或真实密钥的条件下，完成一条可验证的“遥测契约 → 指标 → SLO/错误预算 → 行动”链路。项目只用Python 3标准库：

- 严格校验结构化事件、Trace、Metric label、Collector与遥测治理契约；
- 验证教学用W3C `traceparent`、父子Span和事件—Trace关联；
- 计算RED、USE、p95、SLI、短/长窗口错误预算燃烧率；
- 同时检查质量、安全、费用、Agent步数/token和观测完整性；
- 输出`OK`、`TICKET`或`PAGE`以及可复核理由和证据指纹；
- 用32项离线测试覆盖正常、异常、非法输入与CLI退出码。

项目文件均使用vault限定链接，避免同名文件歧义：

- [[运行监控/03-项目与自测/examples/monitor_audit.py|monitor_audit.py]]：契约验证、指标计算和决策CLI；
- [[运行监控/03-项目与自测/examples/telemetry_windows.json|telemetry_windows.json]]：版本化教学策略、SLO及两个窗口；
- [[运行监控/03-项目与自测/examples/test_monitor_audit.py|test_monitor_audit.py]]：标准库`unittest`测试。

## 环境准备

主要环境是Windows 11、PowerShell 7和当前稳定Python 3。先学习`venv + pip`的标准流程；本项目没有第三方依赖，因此无需安装包。为避免在vault中生成`.venv`，把虚拟环境放到系统临时目录：

```powershell
$projectDir = "X:\path\to\your-vault\Knowledge\AI Agent Engineer\docs\运行监控\03-项目与自测\examples"
$venvDir = Join-Path $env:TEMP "ai-agent-engineer-monitoring-venv"
python -m venv $venvDir
& (Join-Path $venvDir "Scripts\Activate.ps1")
Push-Location $projectDir
```

若本机策略不允许激活脚本，可直接使用`& (Join-Path $venvDir "Scripts\python.exe") ...`。本项目无需网络或API key，也不会写报告文件；不要将虚拟环境复制进知识库。

## 输入契约

顶层JSON分为`policy`、`slo`和`scenarios`。验证器使用精确字段集合：缺字段、未知字段、错误类型、非法枚举、非有限数、重复ID和未排序时间都会失败，避免悄悄忽略拼写错误。

### 事件与Agent信号

每个事件包括：

- `request_id`、带时区时间、`service`、`release_id`和受控`task_type`；
- 技术`status`、`latency_ms`、`trace_id`；
- provider、model snapshot、模型/工具调用数、Agent步数、输入/输出token；
- 可为`true/false/null`的质量标签，`null`明确表示真值未知；
- 安全检查状态、违规结果和估算费用。

脚本会拒绝“未执行安全检查却宣称发现违规”等矛盾。provider/model字段是本地契约示例，不声称对应任何供应商当前SDK字段。

### Trace与传播

每条Trace包含人工构造的教学ID和Span树。验证器检查：

- W3C版本`00`形状、32 个小写十六进制字符且非全零的Trace ID、16 个小写十六进制字符且非全零的父Span ID及`00/01`标志；
- `traceparent`中的Trace ID与记录一致；
- 恰有一个根Span、Span ID唯一、父Span真实存在且无环；
- 根Span的service、release、status和duration与事件可关联。

这些ID不来自真实用户或生产系统。`traceparent`只用于传播关联，不能作为认证凭据。

### 遥测与Collector治理

`metric_label_keys`必须来自有限allowlist，`request_id`等高基数值会被拒绝。遥测策略检查原始内容采集是否关闭、脱敏检查和保留期；Collector信号覆盖accepted、refused、sent、failed、queue和最后导出时龄。资源信号用USE检查CPU利用率、队列饱和和资源错误。

## 指标与行动口径

| 口径 | 本项目定义 | 解释边界 |
| --- | --- | --- |
| RED Rate | 窗口事件数 ÷ 窗口分钟 | 教学事件率，不是流量预测 |
| RED Errors | `status != accepted`比例 | 技术失败，不等于任务质量 |
| RED Duration | 全部教学事件的nearest-rank p95 | 不代表特定监控平台算法 |
| SLI好事件 | `status`合格且延迟不超过SLO目标 | 事件定义和延迟目标已版本化 |
| 燃烧率 | 坏事件比例 ÷ `(1 - SLO目标)` | `1x`表示按允许预算匀速消耗 |
| Trace完整率 | 具有可验证关联Trace的事件比例 | 缺失表示证据不足，不表示请求不存在 |
| 质量通过率 | 已标签事件中的通过比例 | 必须与标签覆盖率并列 |
| 安全违规率 | 已检查事件中的违规比例 | 必须与安全检查覆盖率并列 |
| 成本/Agent结构 | 每请求估算成本、步数、token均值 | 是教学估算，不是真实账单 |

短窗口覆盖场景末尾5分钟，长窗口覆盖完整60分钟。决策优先级为`OK < TICKET < PAGE`：持续的短/长窗口高速燃烧、安全/隐私策略失败、Collector失明或Trace完整率不足可升级到PAGE；延迟、标签、质量、成本、Agent结构和USE调查线可生成TICKET。阈值属于`local-monitor-policy-v2`教学策略，不是Google、Prometheus或OpenTelemetry推荐的通用数值。

## 运行场景

先运行健康窗口：

```powershell
$env:PYTHONDONTWRITEBYTECODE = "1"
python .\monitor_audit.py --scenario stable-window
if ($LASTEXITCODE -ne 0) { throw "stable-window未按预期返回0" }
```

预期首行为`OK: stable-window`，短/长燃烧率为`0.00/0.00`。再运行故障窗口：

```powershell
python .\monitor_audit.py --scenario incident-window
if ($LASTEXITCODE -ne 1) { throw "incident-window未按预期返回1" }
```

预期首行为`PAGE: incident-window`，本地样例的短/长燃烧率为`5.00/2.50`，随后列出SLO、安全、Collector、Trace、质量、成本和资源等判定理由。

CLI退出码契约：

- `0`：所选场景为`OK`；
- `1`：至少一个场景需要`TICKET`或`PAGE`；
- `2`：文件、JSON或数据契约错误。

这让脚本可用于离线检查，但真实PAGE仍需所有者、runbook、路由、抑制和端到端通知验证。

## 运行测试与语法检查

下面三组测试分别覆盖正常解释器、禁用断言和将warning视为错误。测试不依赖生产代码中的`assert`：

```powershell
python -m unittest -v test_monitor_audit
if ($LASTEXITCODE -ne 0) { throw "normal tests failed" }

python -O -m unittest -v test_monitor_audit
if ($LASTEXITCODE -ne 0) { throw "optimized tests failed" }

python -W error -m unittest -v test_monitor_audit
if ($LASTEXITCODE -ne 0) { throw "warnings-as-errors tests failed" }
```

再做不写`.pyc`的最小语法检查：

```powershell
python -c "from pathlib import Path; [compile(p.read_text(encoding='utf-8'), str(p), 'exec') for p in map(Path, ('monitor_audit.py', 'test_monitor_audit.py'))]"
if ($LASTEXITCODE -ne 0) { throw "syntax check failed" }
Pop-Location
```

测试会覆盖严格schema、时间、非有限数、Trace Context、父子Span、环、事件关联、高基数label、Collector与安全矛盾、决策优先级、证据指纹和CLI退出码。通过测试只能证明本地契约与实现一致，不能证明真实监控后端或通知链路可用。

## 动手练习与扩展

1. 新增`ticket-window`，让长窗口超标而短窗口未超PAGE线，断言结果为`TICKET`；
2. 新增一个全部质量标签为`null`的合法窗口，验证输出明确写“任务质量未知”而不是质量为零；
3. 为异步工具Span增加link教学字段，先写失败测试，再扩展精确schema与环检测；
4. 将决策输出序列化为stdout JSON，保留策略/SLO版本和证据指纹，但不要覆盖输入或生成历史产物；
5. 为每条PAGE理由建立离线runbook映射，并测试缺少owner、runbook或恢复条件时不能进入真实通知路由。

## 自测题

1. 为什么HTTP成功且质量标签缺失时，不能把请求记为质量成功？
2. 为什么Trace采样率下降与Metric label基数上升是两个不同问题？
3. 为什么短窗口和长窗口同时越线，比单个瞬时阈值更适合PAGE？
4. 为什么Collector导出时龄过高时，其余“正常”指标也不可信？
5. 为什么`traceparent`格式正确仍不能证明请求已授权？
6. 为什么incident窗口退出码`1`在测试中是成功验证，而不是脚本崩溃？

## 掌握检查

- [ ] 能解释JSON契约中的每个分母、未知值和版本字段；
- [ ] 能从RED症状下钻到USE资源、Trace和结构化事件；
- [ ] 能手算SLI、错误预算和短/长窗口燃烧率；
- [ ] 能区分质量、安全、成本、Collector健康与观测完整性；
- [ ] 能运行32项测试，并理解`-O`与`-W error`验证的风险；
- [ ] 能说明所有ID、阈值和模型字段只是离线教学契约；
- [ ] 能把真实系统仍需验证的采集、传播、后端、通知、隐私和合规边界列清。

## 项目边界与下一步

本项目证明的是：输入契约、关联、指标口径和本地决策可被确定性验证。它没有连接真实SDK、Collector、监控后端、供应商计费、通知链路或用户真值，也没有进行Obsidian阅读视图验证。实战中应先在沙箱接入 [[运行监控/01-可观测性基础/02-仪器化、Collector与关联|仪器化、Collector 与关联]] 的流水线，再用 [[运行监控/02-生产监控/07-事件响应与复盘|事件响应与复盘]] 演练从症状到遏制和恢复的完整链路。
