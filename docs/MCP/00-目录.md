# MCP

## 知识库简介

Model Context Protocol（MCP）是一套让 AI 应用以统一方式连接外部能力与上下文的开放协议。本知识库不把目标停在“会复制一段 server 配置”，而是让初学者能：

- 解释 host、client、server 与双向能力；
- 读懂 JSON-RPC 生命周期和 capability 协商；
- 为 tools/resources/prompts 与 roots/sampling/elicitation 选择正确边界；
- 比较 stdio 与 Streamable HTTP；
- 做授权、安全、调试与版本审查；
- 用离线项目验证消息、schema、错误和实验性 Tasks 的关键规则；
- 用真实 loopback HTTP 往返验证 Streamable HTTP、session 与 OAuth resource boundary。

知识库分两层：

1. **学习路线**：7 篇稳定核心课程与可运行项目，先学这一层；另有 1 篇前沿观察专题，不作为稳定前置。
2. **官方文档参考层**：任务开始前已有的 18 篇本地中文摘要，均采用“核心结论—主要内容—注意事项—相关页面”压缩模板，不是上游正文镜像或完整翻译。它们原则上保持冻结并按需查阅；已确认的问题只做窄范围勘误。

## 在总路线中的位置

MCP 是“单 Agent 与工具”阶段的按需互操作分支。课程目录中的编号只表示展示顺序；只有当项目需要跨进程、跨产品或跨组织标准化交换工具、资源、提示或 client features 时，才需要进入本库。开始前应具备：

- [[API/00-目录|API]]：HTTP、认证、超时与错误；
- [[JSON/00-目录|JSON]]：JSON 与 JSON Schema；
- 若主要目标是暴露或调用 tools，先学 [[Tool Calling（含 Function Calling）/00-目录|Tool Calling（含 Function Calling）]]，理解模型如何选择工具、host 怎样验证与调度；
- 在连接不可信 resource、远程 server 或把协议检查写成回归用例前，完成总路线的[[All of AI#早期安全与评测 Milestone|早期安全与评测 milestone]]。它不是整门 AI 安全与评测体系的前置要求。

MCP 进一步把工具、上下文和 client features 标准化为跨实现的协议边界。它不是 Agent 的规划循环，也不是 [[Agent 核心/00-目录|Agent 核心]] 的普遍前置；可在 Agent 核心前后按互操作需求学习。只使用进程内函数、直接 SDK 或既有私有接口且没有协议互操作需求时，可以暂不学习 MCP。

## 学习目标

- 分清 host、client、server，以及数据层与传输层。
- 用方向矩阵解释 server features 与 client features。
- 正确区分 request、response、notification 和双向 request ID。
- 解释初始化、协议版本选择、capability/sub-capability 与运行/关闭。
- 写出 tool input/output schema，并区分协议错误和工具执行错误。
- 理解 roots、sampling、elicitation 与 Tasks 的控制权和安全边界。
- 比较 stdio、Streamable HTTP 与旧 HTTP+SSE 资料。
- 使用 Inspector 和分层日志定位故障，记录四类版本事实。
- 完成离线数据驱动验证项目，理解它为何不是官方 conformance suite。
- 运行真实 loopback Streamable HTTP，区分 transport、session、OAuth resource policy 与消息 schema。

## 前置知识

- 会读 JSON 对象、数组和最小 JSON Schema。
- 理解 HTTP request/response、status code、bearer token、timeout。
- 能在 Windows 11 / PowerShell 7 运行 `python -B script.py`。
- 知道 stdout 与 stderr 的区别。
- 不要求先会某个 MCP SDK；先学协议，再选实现。

## 核心心智模型

| 能力拥有者 | 主要能力 | 典型请求方向 |
| --- | --- | --- |
| server | tools、resources、prompts、logging、completions | client → server |
| client | roots、sampling、elicitation | server → client |
| 双方/utility | ping、progress、cancellation、实验性 Tasks | 依具体方法与 capability |

capability 表示“该会话声明可用的协议功能”，不等于用户已同意、资源已授权或 server 内容可信。

## 动手实践入口

### 主项目：离线 MCP 教学验证器

从仓库根目录运行：

```powershell
Set-Location ".\docs\MCP" # 进入 MCP 教学项目目录，使后续相对路径可正确解析
python -B .\examples\validate_mcp_messages.py # 运行 54 个离线 JSON-RPC 场景；-B 不生成 __pycache__
python -B .\examples\test_validate_mcp_messages.py # 在普通解释器模式运行全部回归测试
python -B -O .\examples\test_validate_mcp_messages.py # 验证关键协议 gate 没有依赖会被 -O 移除的 bare assert
python -B -W error .\examples\test_validate_mcp_messages.py # 把任何警告升级为失败，及早暴露兼容性问题
python -B -O -W error .\examples\test_validate_mcp_messages.py # 同时覆盖优化模式与严格警告模式的组合
```

当前验收基线：

- 严格 JSON fixture：54 个场景；
- 16 个正向通过、38 个负向按预期拒绝；
- 108 个单元/数据驱动测试；
- 普通与 `-O` 模式、warnings-as-errors 均通过；
- 无网络、密钥、第三方依赖或缓存要求。

### 进阶项目：Loopback Streamable HTTP 与 OAuth 资源边界

先完成离线消息状态机，再运行独立的真实 HTTP boundary：

```powershell
python -B .\examples\streamable_http\loopback_mcp_http.py demo # 运行真实 loopback HTTP 的正常握手、JSON/SSE 与 session 链路
python -B .\examples\streamable_http\loopback_mcp_http.py attack # 运行攻击链路；输出 BLOCK 代表攻击被成功拒绝
python -B .\examples\streamable_http\test_loopback_mcp_http.py # 普通模式运行 80 个 HTTP boundary 回归测试
python -B -O .\examples\streamable_http\test_loopback_mcp_http.py # 优化模式验证 production gate 不依赖 bare assert
python -B -W error .\examples\streamable_http\test_loopback_mcp_http.py # 将资源、线程和 HTTP 警告提升为失败
python -B -O -W error .\examples\streamable_http\test_loopback_mcp_http.py # 组合最严格的解释器与警告检查
```

当前验收基线：真实 loopback POST/GET/DELETE 与 JSON/SSE 往返；CLI 分别输出 `PASS`/`BLOCK`；80 个测试覆盖初始化替代版本、Origin、header/status、Unicode identity/session、PRM 文档/challenge 形状、RFC 8707 resource、scope/tenant/revision、并发、JSON 深度预算、显式恢复、无 cursor 不重放、过期和撤销。详见 [[MCP/学习路线/08-项目-Loopback-Streamable-HTTP与OAuth资源边界|项目课程]]。

> [!warning] 授权能力边界
> 该项目的 HTTP 是真实的，token policy 是离线 fixture。它没有 authorization server、PKCE、验签/JWKS、introspection 或 TLS，不能被描述成“已经实现 OAuth”；其 `http://127.0.0.1` resource 只验证 PRM 形状，不符合 RFC 9728 的 HTTPS 要求。

### 第三项目：真实 SDK 最小互操作

完成两个教学项目后，选择一种官方 SDK：

1. 按 [[MCP/03-使用 MCP 开发/04-构建 MCP 服务器|构建 MCP 服务器]] 做只读、离线最小 server。
2. 用 [[MCP/06-开发者工具/01-MCP Inspector|MCP Inspector]] 验证 initialize 与 tool/resource。
3. 按 [[MCP/03-使用 MCP 开发/05-构建 MCP 客户端|构建 MCP 客户端]] 做最小 client。
4. 记录协议规范、协商版本、SDK、client/server 版本。
5. 再加一个负向 schema、一个 capability 缺失和一个安全测试。

动态 SDK 命令、包名和参数以实际开发时的官方文档为准，不从本路线猜测。

## 掌握标准

- [ ] 能解释一个 host 为何通常为每个 server 建立独立 client。
- [ ] 能不看笔记列出三类 server features 与三类 client features。
- [ ] 能为任一消息说明方向、ID、capability 和生命周期状态。
- [ ] 能说明 `sampling.tools`、elicitation mode 与 Tasks 为什么需要 sub-capability。
- [ ] 能写有必填、枚举、未知字段策略的 input schema 和可验证 output schema。
- [ ] 能区分 JSON-RPC protocol error 与 tool execution error。
- [ ] 能说明 stdio stdout 限制与 Streamable HTTP 的 Origin/session/authorization 风险。
- [ ] 能解释 token passthrough 为什么被禁止，form/url elicitation 的秘密边界是什么。
- [ ] 能按七层法用 Inspector/日志定位问题，并记录四种版本事实。
- [ ] 能运行并解释离线项目的 54 场景与 108 tests。
- [ ] 能运行 loopback 项目的 CLI `PASS`/`BLOCK` 与 80 tests，并解释初始化版本协商、JSON/SSE、Unicode identity/session 和 401/403。
- [ ] 能明确指出离线消息验证器未覆盖真实传输；loopback 项目仍未覆盖真实 OAuth、完整规范与 SDK 互操作。
