# MCP

## 知识库简介

Model Context Protocol（MCP）是一套让 AI 应用以统一方式连接外部能力与上下文的开放协议。本知识库不把目标停在“会复制一段 server 配置”，而是让初学者能：

- 解释 host、client、server 与双向能力；
- 读懂 JSON-RPC 生命周期和 capability 协商；
- 为 tools/resources/prompts 与 roots/sampling/elicitation 选择正确边界；
- 比较 stdio 与 Streamable HTTP；
- 做授权、安全、调试与版本审查；
- 用离线项目验证消息、schema、错误和实验性 Tasks 的关键规则。

知识库分两层：

1. **学习路线**：本轮自行组织的 6 篇初学者课程与可运行项目，先学这一层。
2. **官方文档参考层**：任务开始前已有的 18 篇官方资料整理。本轮保留正文不改，用于按需查阅和真实 SDK 实践。

> [!info] 动态事实边界
> 本路线按 2026-07-14 获取的第一方资料编写。当日官方版本页与规范站点把 `2025-11-25` 标为 latest；真实实现仍必须核对 host、client、server、SDK 支持，并以 initialize 实际协商版本为准。URL elicitation、sampling tool use 和 Tasks 等较新边界不应被假定为所有实现均已支持；Tasks 在该规范中仍标记为实验性。

## 在总路线中的位置

MCP 是“单 Agent 与工具”阶段的第 2 个知识库。建议先学：

- [[API/00-目录|API]]：HTTP、认证、超时与错误；
- [[JSON/00-目录|JSON]]：JSON 与 JSON Schema；
- [[Tool Calling（含 Function Calling）/00-目录|Tool Calling（含 Function Calling）]]：模型如何选择工具、host 怎样验证与调度。

MCP 进一步把工具、上下文和 client features 标准化为跨实现的协议边界。它不是 Agent 的规划循环；学完后进入 [[Agent 核心/00-目录|Agent 核心]]。

## 学习目标

- 分清 host、client、server，以及数据层与传输层。
- 用方向矩阵解释 server features 与 client features。
- 正确区分 request、response、notification 和双向 request ID。
- 解释初始化、协议版本选择、capability/sub-capability 与运行/关闭。
- 写出 tool input/output schema，并区分协议错误和工具执行错误。
- 理解 roots、sampling、elicitation 与 Tasks 的控制权和安全边界。
- 比较 stdio、Streamable HTTP 与旧 HTTP+SSE 资料。
- 使用 Inspector 和分层日志定位故障，记录四类版本事实。
- 完成离线数据驱动验证项目，理解它为何不是官方 conformance suite。

## 前置知识

- 会读 JSON 对象、数组和最小 JSON Schema。
- 理解 HTTP request/response、status code、bearer token、timeout。
- 能在 Windows 11 / PowerShell 7 运行 `python -B script.py`。
- 知道 stdout 与 stderr 的区别。
- 不要求先会某个 MCP SDK；先学协议，再选实现。

## 核心心智模型

| 能力拥有者 | 主要能力 | 典型请求方向 |
| --- | --- | --- |
| server | tools、resources、prompts、logging、completions | client → server |
| client | roots、sampling、elicitation | server → client |
| 双方/utility | ping、progress、cancellation、实验性 Tasks | 依具体方法与 capability |

capability 表示“该会话声明可用的协议功能”，不等于用户已同意、资源已授权或 server 内容可信。

## 动手实践入口

### 主项目：离线 MCP 教学验证器

```powershell
Set-Location "X:\path\to\your-vault\Knowledge\AI Agent Engineer\docs\MCP"
python -B .\examples\validate_mcp_messages.py
python -B -W error .\examples\test_validate_mcp_messages.py
python -B -O -W error .\examples\test_validate_mcp_messages.py
```

当前验收基线：

- 严格 JSON fixture：33 个场景；
- 10 个正向通过、23 个负向按预期拒绝；
- 58 个单元/数据驱动测试；
- 普通与 `-O` 模式、warnings-as-errors 均通过；
- 无网络、密钥、第三方依赖或缓存要求。

### 第二项目：真实 SDK 最小互操作

完成离线项目后，选择一种官方 SDK：

1. 按 [[MCP/03-使用 MCP 开发/04-构建 MCP 服务器|构建 MCP 服务器]] 做只读、离线最小 server。
2. 用 [[MCP/06-开发者工具/01-MCP Inspector|MCP Inspector]] 验证 initialize 与 tool/resource。
3. 按 [[MCP/03-使用 MCP 开发/05-构建 MCP 客户端|构建 MCP 客户端]] 做最小 client。
4. 记录协议规范、协商版本、SDK、client/server 版本。
5. 再加一个负向 schema、一个 capability 缺失和一个安全测试。

动态 SDK 命令、包名和参数以实际开发时的官方文档为准，不从本路线猜测。

## 掌握标准

- [ ] 能解释一个 host 为何通常为每个 server 建立独立 client。
- [ ] 能不看笔记列出三类 server features 与三类 client features。
- [ ] 能为任一消息说明方向、ID、capability 和生命周期状态。
- [ ] 能说明 `sampling.tools`、elicitation mode 与 Tasks 为什么需要 sub-capability。
- [ ] 能写有必填、枚举、未知字段策略的 input schema 和可验证 output schema。
- [ ] 能区分 JSON-RPC protocol error 与 tool execution error。
- [ ] 能说明 stdio stdout 限制与 Streamable HTTP 的 Origin/session/authorization 风险。
- [ ] 能解释 token passthrough 为什么被禁止，form/url elicitation 的秘密边界是什么。
- [ ] 能按七层法用 Inspector/日志定位问题，并记录四种版本事实。
- [ ] 能运行并解释离线项目的 33 场景与 58 tests。
- [ ] 能明确指出课程验证器未覆盖真实传输、完整规范、授权、并发和互操作。
