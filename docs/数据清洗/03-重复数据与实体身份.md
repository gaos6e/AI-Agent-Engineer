# 重复数据与实体身份

## 本节目标

区分重复导入、同实体冲突、真实重试与语义近重复，并让去重规则保留身份、来源、权限和可追溯关系。

## 三种“重复”

- **完全重复**：所有字段相同，常来自重复导入。
- **同实体冲突记录**：主键相同但字段不同，可能是更新、重试或数据损坏。
- **语义近重复**：文本稍改、截图裁切或同一文档不同版本，不能用整行相等发现。

去重前必须定义实体和事件。两个相同请求可能是用户真正重试，也可能是采集端重放；直接删除会改变失败率和流量统计。

## 键与幂等性

稳定主键用于识别同一实体；事件 ID 用于识别同一次发生。若外部 API 会重试，写入端应使用幂等键，清洗端再根据事件 ID 识别重放。

出现主键冲突时不要“保留最后一条”就结束。先确定排序字段可靠且含时区，再记录被替换版本、选择规则和冲突数量。

还要预先定义“首次出现但无效”的 ID 是否占有身份。本库项目选择占有：后一条同 ID 即使字段有效也进入重复报告，避免悄悄用后到数据改写历史；其他业务可以选择不同策略，但必须写进契约和测试。

## 文本和 RAG 去重

文档可按内容哈希发现完全重复；近重复可用 n-gram/Jaccard 或 Embedding 候选，再人工检查。文档去重应保留来源与权限：内容相同但访问控制不同，不能只留一个公共副本。

数据划分必须按原文档或实体分组，否则同一内容的 chunk 会跨训练/测试，造成泄漏。

## 同源关系、权限与撤回

“内容可合并”不等于“来源可以消失”。即使用一个规范化内容对象节省存储，也要保留每个 `source_id`、版本、`access_scope` 与保留—合并映射；否则无法回答“这段文本由谁授权、哪些用户可见、它来自哪个版本”。对于冻结评测，`group_id` 应来自可解释的源实体（文档、会话、事故、代码仓库或时间窗），并在看标签、模型分数或未来结果前确定。

若清洗后的样本要进入评测或 Benchmark，显式冻结 `group_id → family_id` 映射：`group_id` 是上游实体/同源边界，`family_id` 是评测 case 的防泄漏分组标识，二者可一一对应也可多对一，但不能由标签、模型输出或随机行号临时生成。术语与 split 规则以[[评测体系/01-基础与设计/02-用例数据集与分层|用例数据集与分层]]和[[Benchmark设计/01-基础与设计/03-数据划分泄漏与污染|数据划分泄漏与污染]]为准。

收到来源更正、删除或权限收回时，不能只从当前表删一行。应以 lineage 找到派生的规范记录、chunk、索引/Embedding、缓存、训练或评测 split、质量报告和公开下载包，先停止新的消费，再按发布政策重建、撤回或标记失效。哈希只能帮助定位字节内容，不能替代这条来源—派生产物映射。

## 验证问题

- 去重前后行数、实体数和标签分布变化多少？
- 被删除记录能否追溯到保留记录？
- 同 ID 冲突是否被单独报告？
- 重复率突然升高是否表示上游重试风暴？

## 练习

为工具调用日志设计 `run_id`、`call_id`、`attempt` 和 `idempotency_key`。说明“同一调用的第二次尝试”在任务成功率、调用量和费用统计中分别如何计算。

## 掌握检查

- [ ] 我能定义实体键、事件键、尝试序号和幂等键的不同职责。
- [ ] 我不会仅凭内容哈希合并权限、来源或版本不同的文档。
- [ ] 我能报告去重前后行数、实体数、标签分布和保留—删除映射。
- [ ] 我会按实体/原文档分组划分数据，防止近重复跨训练与测试。
- [ ] 我能为一个来源撤回列出所有需要复核的派生产物，而不是只删除最终表的一行。
