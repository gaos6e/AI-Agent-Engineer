# 项目：可降级规则重排器

## 项目目标

本项目不训练神经模型，而是实现生产适配层最容易被忽略的部分：

```text
strict candidate fixture
        ↓
tenant / ACL / status / effective-time 防御复核
        ↓
safe candidate window
        ↓
透明 rule provider（可模拟五种状态）
        ↓
exact output-ID / finite-score / feature 校验
        ↓
正常重排 or 同一安全窗口 first-stage fallback
        ↓
canonical cap + MRR/nDCG/Precision + 安全报告
```

规则分只验证 orchestration；它不是 Cross-Encoder、LTR 或 LLM 的质量替代。

## 项目文件

- [[Reranking/examples/reranker-fixture.json|reranker-fixture.json]]：query、9 个候选、graded qrels、must-not 与设置；
- [[Reranking/examples/toy_reranker.py|toy_reranker.py]]：严格加载、过滤、评分、输出校验、fallback 与指标；
- [[Reranking/examples/test_toy_reranker.py|test_toy_reranker.py]]：26 项测试。

Fixture 解析拒绝重复 JSON key、NaN/Infinity、未知/缺失字段、错误类型、重复 ID/rank、不连续 first rank、非法日期、错误有效期、不可访问 qrels，以及仍可访问的 must_not_return。

## 候选与质量上限

Query 是“退款审核通过后多久到账”，as-of=2026-07-14。9 个 first-stage 候选中：

- 6 个满足 alpha/guests、published 与有效期；
- d-07 因 wrong tenant 排除；
- d-08 因 expired 排除；
- d-09 因 ACL denied 排除。

Qrels 有 4 个正例：申请入口 grade 1、到账正文 grade 3、FAQ grade 2、延迟处理 grade 2。安全候选的 first rank 分别为 2/4/5/6：

- window=6：4/4，Candidate Recall=1；
- window=3：只含 rank 2，Candidate Recall=0.25。

这让窗口上限可被直接验证，而不是只在正文里宣称。

## 透明评分

Analyzer 做 NFKC/casefold、ASCII 字母数字串与中文重叠二字符。规则特征为：

- title_coverage；
- body_coverage；
- exact_phrase；
- score = 2 × title coverage + body coverage + exact phrase。

这组权重只是确定性 fixture 设计，不可推广。正常输出仍要通过与外部模型相同的契约：

- ID 必须等于整个输入 window 的 exact set；
- 每个 ID 恰好一次；
- score/features 必须是有限数值；
- 未知、遗漏、重复、空结果均 invalid；
- 平局按 first-stage rank，再按 ID。

## 正常结果

默认 window=6、top-n=3、每 canonical source 最多 2 条：

| 路径 | 排名 | MRR@3 | nDCG@3 | Precision@3 |
| --- | --- | ---: | ---: | ---: |
| First stage | d-01, d-02, d-03 | 0.500000 | 0.060708 | 0.333333 |
| Rule rerank | d-04, d-05, d-06 | 1.000000 | 1.000000 | 1.000000 |

security_violations 必须为空。结果只证明该手工规则在当前 9 条 fixture 上按设计工作，不代表真实模型收益。

## 失败模式

CLI 提供：

| failure | 模拟 | fallback_reason |
| --- | --- | --- |
| none | 完整合法结果 | null |
| timeout | provider 超时 | timeout |
| error | provider/server 错误 | provider_error |
| empty | 空 list | invalid_output |
| malformed | 重复 ID、漏 ID | invalid_output |

四个失败模式都应：

- rerank_applied=false；
- final 与 safe first-stage 完全一致；
- 排名 d-01, d-02, d-03；
- security_violations 为空；
- wrong tenant、expired、ACL denied 不会被 fallback 带回。

## 运行正常与失败路径

从 vault 根目录运行：

```powershell
$env:PYTHONDONTWRITEBYTECODE = '1'
$script = '.\Knowledge\AI Agent Engineer\docs\Reranking\examples\toy_reranker.py'
$fixture = '.\Knowledge\AI Agent Engineer\docs\Reranking\examples\reranker-fixture.json'

python -B -W error $script --fixture $fixture
if ($LASTEXITCODE -ne 0) {
    throw "正常路径失败，退出码：$LASTEXITCODE"
}

foreach ($mode in @('timeout', 'error', 'empty', 'malformed')) {
    python -B -W error $script --fixture $fixture --failure $mode
    if ($LASTEXITCODE -ne 0) {
        throw "失败模拟 $mode 未产生受控报告，退出码：$LASTEXITCODE"
    }
}
```

Provider 故障是被适配层成功处理的业务状态，因此 CLI 返回 0 并输出 fallback 报告；fixture/参数错误才返回 2。

## 关键实验

### 实验 A：窗口上限

```powershell
python -B -W error $script --fixture $fixture --candidate-window 3 --output-top-n 3
```

确认 candidate_recall_at_window=0.25，grade-3 到账正文不在窗口。此时重排规则再好也无法让它进入 top-3。

### 实验 B：Canonical cap

```powershell
python -B -W error $script --fixture $fixture --max-per-canonical 1
```

默认 d-04/d-05 属于同一 canonical document；cap=1 后第二条被跳过，由 d-06 和 d-02 补位。比较 item-level nDCG 与 source/evidence coverage，不能只宣称“更多样”。

### 实验 C：输出攻击

在测试中构造 unknown ID、重复 ID、漏 ID、NaN score 和非数值 feature，确认都触发 OutputContractError/fallback，且不进入 final ranking。

### 实验 D：接入真实 Cross-Encoder

替换 simulate_provider 时保留：

1. 相同 safe window；
2. request/response schema 与 exact ID 校验；
3. model/tokenizer/input-template revision；
4. token/truncation、batch、deadline 和耗时；
5. 正常与所有失败路径；
6. first-stage/rule/新模型、关键切片和端到端对比。

## 与生产系统的差距

本项目没有神经模型、GPU/队列、真实网络、batching、tokenizer/truncation、rate limit、重试/断路、缓存、分布式 trace、真实鉴权或敏感日志治理。透明 n-gram 规则只适合教学，不应被包装为 relevance 服务。

## 项目验收

- [ ] Candidate recall 上限能随 window 改变被复现。
- [ ] 硬过滤在模型前，fallback 只用 safe window。
- [ ] 输出 ID exact、唯一、完整，score/features 有限。
- [ ] 四类故障均产生确定、可观测 fallback。
- [ ] Normal/fallback first-stage、model、final provenance 分离。
- [ ] Canonical cap 从后续候选补位。
- [ ] 正常与 -O 模式各 26 项测试通过。
- [ ] 无凭据、网络、缓存或第三方依赖。
- [ ] 当前规则指标不外推为模型性能。

## 自测

1. 为什么 window=3 时 grade-3 正例不可能被重排到首位？
2. 为什么空 model output 不能自动解释为“全部不相关”？
3. 为什么 fallback 必须复用 safe window，而不是原始未过滤列表？
4. 为什么 output contract 要求 exact ID set？
5. Canonical cap 如何同时改善多样性又可能降低 item nDCG？
6. 接入真实 Cross-Encoder 后必须新增哪些版本与延迟字段？
