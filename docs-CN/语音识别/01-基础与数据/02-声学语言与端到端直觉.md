# 声学、语言与端到端直觉

## 本节目标

不推导复杂模型，也能解释声学证据、语言约束和端到端解码的作用。

## 模块化系统的直觉

传统 ASR 常把问题拆开：

- **声学模型（acoustic model）**：从音频特征估计音素或其他声学单位。
- **发音词典（lexicon）**：连接词和发音序列。
- **语言模型（language model）**：在多个听起来接近的候选中偏好更合理的词序列。
- **解码器（decoder）**：综合分数搜索结果。

这使模块可单独替换和诊断，但构建词典、对齐和搜索较复杂。

## 端到端系统的直觉

端到端方法直接学习“音频序列到文本序列”。常见训练/解码思想包括：

- **CTC**：允许大量空白标记，再合并重复符号，解决音频帧比字符多的问题。
- **注意力编码器—解码器**：编码音频表示，解码器逐步生成文本并关注相关片段。
- **Transducer/RNN-T 类方法**：联合时间推进和符号输出，适合增量解码。

它们不是“完全没有语言知识”；训练文本分布、tokenizer 和解码仍会形成语言偏好。端到端也不等于自动获得可靠时间戳、说话人或置信度。

Whisper 原始论文描述了大规模弱监督、多任务序列到序列训练；官方仓库是理解其安装、模型尺寸和长音频处理取舍的一份具体材料。它不应被外推成“所有 ASR 都使用相同窗口、支持相同语言、给出相同时间戳或适用于相同风险场景”。

无论系统看似“端到端”还是模块化，生产输出都由一组可变组件共同决定：

| 记录项 | 为什么不能省略 |
| --- | --- |
| `model_id` / `model_revision` | 同名模型、权重或托管快照可能改变 |
| `frontend_revision` | 解码、下混、重采样和特征提取会改变输入 |
| `decode_config_revision` | 语言提示、词表偏置、温度、VAD 阈值等会改变结果 |
| `normalization_revision` | 同一 raw transcript 的展示和 WER 可能随规则改变 |
| `segment_id + revision` | 流式结果会修订，不能只凭字符串覆盖历史 |

这些字段能帮助排查，而不是让模型输出自动变成真相、授权或身份凭据。

## 选择与排查

选择时先决定处理路径，再比较目标语言、口音/场景、实时性、硬件、许可、可部署位置和可解释输出：

| 路径 | 适合 | 主要工程问题 | 不应推断 |
| --- | --- | --- | --- |
| 本地/离线批处理 | 数据不能离开受控环境、可排队任务 | 模型/codec 依赖、容量、升级回归 | “离线”自动等于无风险或更准确 |
| 托管文件转录 | 有界文件、异步归档 | 上传合同、数据处理、大小/时长、重试 | 文件 API 能直接给实时 partial |
| 实时流 | 字幕、通话、交互 | 背压、turn、修订、断线恢复 | “低延迟”自动等于可执行用户意图 |

错误分析可以问：

- 专有名词错：词表/上下文偏置是否受支持、规则是否越界？
- 静音处幻觉：VAD、分段、停止条件和无语音判定是否合理？
- 长音频重复或缺句：窗口重叠与合并逻辑是否稳定？

## 练习与自测

- 用一句话分别说明声学模型和语言模型的职责。
- 为什么“语言模型更强”也可能改坏人名？它会偏好常见序列，稀有名字可能被替换。
- 端到端模型是否免除切分和评测？否，这些是系统层职责。
