# AI地图与核心术语

## 本节目标

学完本节，你能把常见 AI 术语放进同一张图，并能判断一个需求究竟是规则自动化、机器学习、LLM 应用还是 Agent 系统。

## 从“系统做了什么”理解 AI

**稳定事实**：OECD 对 AI 系统的定义强调：机器系统针对显式或隐式目标，从输入中推断如何生成预测、内容、建议或决策，并可能影响物理或虚拟环境；不同系统的自主性和部署后适应性可以不同。这里的关键词不是“像人”，而是：

1. 有输入，例如文字、图片、传感器数据；
2. 通过某种模型或知识规则进行推断；
3. 产生会被人或其他系统使用的输出；
4. 在明确的使用情境中带来效果和风险。

因此，“AI”不是一个单一算法。一个产品通常还包含数据库、接口、权限、界面、日志和人工流程。模型只是 AI 系统的一部分。

## 六个容易混淆的概念

| 概念 | 初学者直觉 | 典型产物 | 例子 |
| --- | --- | --- | --- |
| 人工智能（AI） | 让机器完成通常需要推断、感知或决策的任务的总称 | AI 系统 | 光学字符识别（OCR）、推荐、语音助手 |
| 机器学习（ML） | 不逐条写死规则，而是从数据中学习模式 | 预测模型 | 垃圾邮件分类、销量预测 |
| 深度学习（DL） | 使用多层神经网络的机器学习方法 | 神经网络模型 | 图像识别、语音识别 |
| 生成式 AI | 生成文字、图像、音频、视频或代码 | 新内容 | 文本摘要、图像生成 |
| 大语言模型（LLM） | 在大量文本或代码上训练、处理 token 序列的语言模型 | 文本或结构化输出 | 问答、改写、代码辅助 |
| Agent | 围绕目标反复观察、决定、行动并读取反馈的系统 | 任务执行轨迹与结果 | 查库存后生成订单草稿 |

关系可粗略画成：

```text
AI
├─ 知识/规则等方法
└─ 机器学习 ML
   └─ 深度学习 DL
      └─ 大量现代生成式模型与 LLM

Agent 系统 = 模型（可能是 LLM，也可能不是）
           + 目标 + 状态 + 工具 + 控制策略 + 反馈
```

这不是产品分类的严格集合图。例如一个 Agent 可以调用规则引擎、搜索模型和 LLM；生成式 AI 也不限于语言。

## AI 不等于自动化，Agent 也不等于聊天框

一个脚本每天 9:00 把文件复制到备份目录，它是**确定性自动化**：相同条件通常执行相同步骤，不需要模型推断。它可能很有价值，但没有必要叫 AI。

一个聊天模型回答一次问题，是一次 **LLM 调用**。只有当系统维护任务状态、选择或调用工具、读取执行反馈，并根据反馈继续推进目标时，才更接近工程语境中的 Agent。

> **工程建议**：选择能满足目标的最简单方案。规则足够时用规则；步骤固定时用工作流；只有路径确实需要依据中间反馈动态选择，并且风险可控时，才增加 Agent 自主性。

## 用四个问题识别系统类型

面对“自动处理客户邮件”的需求，依次问：

1. **输出是否可以用稳定规则得到？** 如果只按发件人域名归档，规则即可。
2. **是否需要从历史样本学习分类边界？** 若要判断多种意图，可能需要 ML 分类器。
3. **是否需要生成或理解开放文本？** 若要概括诉求和生成答复草稿，可能需要 LLM。
4. **是否要根据结果选择动作并操作外部系统？** 若要查订单、决定下一步、创建工单并验证状态，才考虑 Agent 或受控工作流。

“用了 AI”从来不是验收标准。验收应是“高优先级邮件漏分率不超过约定阈值”“所有退款动作都需人工批准”等可观察结果。

## 练习：给需求分类

把下列任务标为“规则 / ML / LLM / 工作流 / Agent”，允许多选，但必须解释理由。

1. 将文件名中的日期统一为 `YYYY-MM-DD`。
2. 根据历史交易识别异常支付。
3. 把一段长说明改写成三条要点。
4. 先提取发票字段，再检查总额，缺字段时通知人工。
5. 读取故障描述，查询监控与知识库，根据结果选择诊断命令，但重启服务前请求批准。

参考思路：1 适合规则；2 通常是 ML；3 是 LLM；4 是固定工作流，字段提取可使用 OCR/ML/LLM；5 有动态工具选择和反馈循环，更接近受控 Agent。分类不是目的，说明“为何需要模型、为何需要自主性”才是目的。

## 自测

1. ML 与 DL 是并列关系还是包含关系？
2. 为什么“模型”不等于“AI 系统”？
3. 一个会调用一次搜索 API 的聊天应用一定是 Agent 吗？
4. 为什么确定性规则有时比 LLM 更好？

参考答案：

1. 通常把 DL 视为 ML 的一个方法家族。
2. 系统还包含数据、接口、权限、人、流程与运行环境；风险也来自这些交互。
3. 不一定。若没有目标驱动的状态、决策与反馈循环，它可能只是增强过的单次调用。
4. 规则更容易解释、复现和测试；当需求稳定明确时，模型的非确定性只会增加成本和风险。
