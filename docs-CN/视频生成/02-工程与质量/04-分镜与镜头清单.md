# 分镜与镜头清单

## 本节目标

把一句创意拆成能单独生成、单独失败、单独重试和单独验收的镜头任务。

## Storyboard 与 shot list 的区别

**Storyboard（分镜板）**用草图或参考帧说明画面和叙事节奏；**shot list（镜头清单）**是可执行表格，记录每个镜头的编号、起止时间、景别、主体、动作、场景、相机、连续性、音频和验收。没有画画能力也可用文字框和简单示意图做 storyboard。

先写故事节拍（beat）：开场要建立什么、转折发生什么、结尾留下什么状态。再把每个节拍拆成一个主动作的短镜头。转场不是用来隐藏逻辑缺口的特效；切镜前后要有合理的方向、位置和状态衔接。

## 镜头清单最小字段

| 字段 | 问题 |
| --- | --- |
| `shot_id` | 是否稳定、唯一，可用于文件名和重试？不要把它混同于需求 `source_revision`、镜头 `transform_id` 或发布 `release_id`。 |
| 时间 | 在总时间线何时开始与结束？ |
| 画面 | 主体、动作、场景、景别、机位、运镜是什么？ |
| 条件 | 使用哪些已授权首帧、角色或物体参考？每项是否有 `asset_id`、`source_revision`、`acl_reference`、用途限制和对象级授权/ACL？ |
| 连续性锚点 | 颜色、服装、道具位置、运动方向、光线状态是什么？ |
| 声音 | 旁白、对白、环境音或静音的意图？ |
| 验收 | 哪些可观察条件必须满足？ |
| 后备方案 | 失败时缩短、拆分、静帧加运动还是人工拍摄？ |

## 连续性 Bible

为重复主体建立小型“连续性手册”：固定名称、外观特征、色板、比例、道具、左右方向、环境光和禁止变化。每个镜头只引用需要的锚点，避免提示过长。镜头生成后由人更新“实际选中版本”的状态，并记录对应 `source_revision` 与 `transform_id`；未被选中的候选不能自动成为下一镜头事实，更不能自动进入发布链。

## 从分镜到提示

不要把整个脚本直接当提示。每个 shot prompt 只写本镜头可见内容，并包含结束状态。例如：`S02，中景跟随；机器人从画面左侧走到桌边，始终拿着蓝杯；暖色顶光；镜头结束时杯子仍在右手，桌面为空。` 旁白和字幕放入独立轨道，避免视觉提示与声音脚本互相污染。

## 常见错误与排查

- 时间线有空隙或重叠：在提交前做数值验证。
- 镜头 ID 随顺序修改：ID 稳定，另有 `sequence_order`。
- 参考资产未声明用途：每个输入附权利、允许使用范围、对象级授权/ACL 与撤权/删除传播目标。
- 连续性写成“保持一致”：列出具体锚点，才能检查。
- 全片一次生成：短镜头工作流更容易局部重试和人工控制。

## 练习与自测

把“雨夜，一辆车到站，乘客下车进入大厅”拆成 4 个镜头。为车、乘客、雨势和运动方向各写一个连续性锚点；再指出一个可在后期完成、不应强求模型生成的元素。
