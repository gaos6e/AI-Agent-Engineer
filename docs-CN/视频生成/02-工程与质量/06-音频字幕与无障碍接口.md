# 音频、字幕与无障碍接口

## 本节目标

把旁白、对白、音乐、音效、字幕和描述性文本作为独立、可同步、可替换的交付物，而不是生成视频的附属品。

## 先设计声音层

常见轨道包括对白/旁白、环境音、动作音效、音乐和静音。为每段记录起止时间、文本、说话者、语言、情绪、响度目标和权利来源。若生成模型可同时产声，也要把音轨单独检查：语义、身份、嘴形同步、噪声、音乐权利和不期望台词。无法稳定满足时，选择后期配音与音效。

音乐“免版税”不等于无条件免费；许可证可能限制平台、地域、改编或商业用途。合成人声涉及身份与同意，不能仅因声音是 AI 输出就忽略冒充和披露风险。

## 字幕是时间数据

WebVTT 是 W3C 定义的时间对齐文本格式，可用于字幕、章节和元数据。最小文件以 `WEBVTT` 开头，每个 cue 有开始、结束与文本：

```text
WEBVTT

00:00.500 --> 00:02.800
机器人把蓝色杯子放到桌面。
```

2026-07-22 核对的 W3C 页面仍是 2026-05-20 Candidate Recommendation Draft，且明确属于可更新的草案。项目要选择目标播放器真正支持的子集并验证。字幕应 UTF-8，时间不倒退、不越界、不过度重叠；自动转写后由人校对人名、术语、数字和标点。

## 可访问性不仅是字幕

字幕面向听不到音频的用户，需要包含有意义的非语音信息，如 `[门铃声]`。**音频描述**面向看不到关键画面的用户，在对白空隙描述重要动作。避免把关键信息只放在颜色或快速闪现文字中；检查对比度、安全区域和阅读速度。不同平台有自己的交付规范，应在发布目标确定后复核。

## 音画同步接口

时间线使用同一零点与时间基准。旁白脚本不能只引用“镜头三附近”，而要引用 `shot_id`、对应 `source_revision` 和时间区间。重新剪辑导致镜头时长变化时，字幕和音频需要重新验证，并记录新的 `transform_id`；批准对外版本再绑定 `release_id`。锁定画面（picture lock）后再做最终混音与字幕定时，可减少反复返工。

字幕、旁白和音频描述可转述外部事实，但合成画面或合成声音本身不证明这些事实。若对外主张新闻、医疗、科学或身份信息，应把文字脚本关联到独立证据和 `release_id`，只有证据足够时才标为 `evidence_supported`。

## 常见错误与排查

- 自动字幕当最终稿：人工对照原脚本和成片。
- 字幕越过剪辑点：按语义与画面边界重切 cue。
- 只检查音频存在：还要检查内容、同步、权利和可理解度。
- 把烧录字幕当唯一版本：同时保留独立文本轨，便于无障碍和多语言。
