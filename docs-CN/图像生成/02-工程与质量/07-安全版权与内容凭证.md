# 安全、版权与内容凭证

## 本节目标

在生成前确认输入权利和预期用途，在生成后保留来源与人工决策记录；不把技术元数据误当成真伪保证。

## 三个时间点的风险控制

**请求前**：确认参考图授权、人物肖像/同意、商标与机密素材范围；识别冒充、欺骗、性化未成年人、仇恨、违法和高风险误导用途；必要时拒绝。**生成后**：检查人物、敏感属性、文字、Logo、相似角色和不实场景；记录评审者与决定。**发布时**：按场景披露合成性质，保留来源记录，遵守平台和适用法律。

这里的版权说明不是法律意见。不同法域、合同和用途不同。以美国版权局 2025 年报告为例，AI 辅助并不自动排除版权，但纯粹提供提示并不足以当然取得对输出的版权保护；项目应记录人类的选择、编排和修改，并向合格法律人员核对实际问题。

## 人物与身份

真实人物输入需要明确同意、用途、保存期限和撤回/删除渠道。即使“只是内部演示”，泄露、误用和二次训练风险仍存在。不能用描述绕过供应商的人脸或公众人物限制。儿童、医疗、情色、政治或声誉场景采用更严格审批。撤回或删除不是只删原文件：需要沿 `asset_id → transform_id → release_id` 让候选、派生图、缓存、缩略图、搜索索引和公开链接按保存/审计规则失效。

## C2PA 与内容凭证

C2PA 定义可签名的 manifest，把创作工具、编辑动作和资产哈希等 provenance（来源与变更历史）与媒体关联。2026-07-22 核对的规范为 **2.4（2026 年 4 月）**。内容凭证可帮助验证签名声明、资产完整性和信任状态；它不对声明内容作真伪价值判断，不能证明画面事件发生过，也不能阻止截屏、剥离元数据或虚假声明。验证失败应区分“没有凭证”“凭证损坏”“签名不可信”，不能一律判假。

内容凭证不是对象级授权/ACL，也不是版权或肖像同意的替代物。ACL 决定某个主体能否为某用途读取或评分；权利记录说明为何可以处理；C2PA 记录可验证的来源声明。三者要分开实现，避免“凭证有效”被误判为“可以发布”。

最低来源记录可包括：

| 字段 | 用途与边界 |
| --- | --- |
| `asset_id` 与 `source_revision` | 定位输入对象及处理时实际采用的版本；哈希用于完整性校验，不替代授权。 |
| `transform_id` | 关联一次生成、编辑、转码或排版变换及其参数/操作者。 |
| `release_id` | 关联被批准的发布候选、披露方式、审核决定和发布位置。 |
| 对象级授权/ACL | 在预览、评分、模型调用和发布前分别核验主体、用途、期限与范围。 |
| 撤权/删除传播 | 记录受影响的候选、派生物、缓存、索引和链接，以及允许保留的最小化审计事件。 |

不要把真实姓名、原始人脸、完整提示或合同全文塞进公开 manifest；隐私与可验证性要平衡。

## 事实性声明的证据边界

生成图可以作为创作、示意、界面原型或被明确披露的合成内容，但它本身不为新闻、医疗、科学或身份主张提供外部事实证据。若发布文本作出可核验的外部陈述，应把陈述关联到独立来源、审核记录和 `release_id`，且只在证据足以支持时标为 `evidence_supported`。内容凭证、模型自述、相似度分数和人工“看起来合理”的判断都不能单独越过这一边界。

`evidence_bound` 与 `evidence_supported` 不同：前者表示提示、脚本或审核任务只能引用已批准的证据集合，限制的是输入范围；后者表示某个将要发布的外部陈述已被足够的独立证据支持，限制的是发布结论。把任务做成 `evidence_bound` 有助于防止凭空补充，但不自动使生成输出或任何陈述 `evidence_supported`。

## 威胁建模

问四件事：谁可能受害、攻击者能控制哪些输入、错误输出如何传播、哪些证据能追溯。常见威胁包括提示注入式素材说明、恶意参考图、生成冒充、移除凭证、审核疲劳、ACL 过宽、撤权后缓存残留和公开存储桶泄露。NIST AI 600-1 建议在全生命周期映射、测量和管理生成式 AI 风险，可用来组织风险台账，但不是一次性勾选表。

## 常见误区

- “有水印就安全”：水印可被裁掉，且不替代授权与审查。
- “C2PA 证明是真的”：它证明签名声明的来源链，不证明事件真实。
- “C2PA 或哈希等于授权”：它们不替代对象级授权/ACL、合同、肖像同意或发布审批。
- “公开网上的图都能当参考”：可访问不等于有使用权。
- “供应商过滤器负责全部”：应用方仍要根据用途做输入与输出治理。
