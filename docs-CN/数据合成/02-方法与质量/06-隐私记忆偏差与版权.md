# 隐私、记忆、偏差与版权

## 本节目标

识别合成数据对隐私、训练记忆、群体偏差、知识产权和内容来源带来的风险，并选择与声明相匹配的控制与人工审批。

## 直觉解释

生成的新句子不一定“来自无中生有”。模型可能复现训练片段，统计生成器可能保留稀有个体组合，提示中也可能直接含真实信息。与此同时，生成器会继承或放大来源分布和安全过滤的偏差。

## 核心概念

- **memorization**：模型保留并在特定条件下复现训练实例或长片段。
- **re-identification / linkage**：把看似去标识数据与其他信息结合后关联到个人。
- **differential privacy（DP）**：量化单个记录加入或移除对输出分布影响的形式框架；保证取决于定义、参数、实现与组合记账。
- **privacy attack**：成员推断、属性推断、记录链接或相似性检查等经验风险测试。
- **bias amplification**：生成分布使原有群体差异、刻板模式或缺失更严重。
- **provenance and license**：来源、使用授权、衍生限制与发布条件。
- **synthetic-content transparency**：对生成内容的来源、处理和标识进行记录。

## 先写威胁模型，再选“隐私指标”

威胁模型至少写：要保护谁、攻击者能看到合成数据/生成器/统计查询中的哪些内容、攻击者可能拥有哪些辅助数据、失败后造成什么伤害。成员推断、属性推断、稀有组合链接和逐字记忆回答的问题不同，单一距离或 PII 正则不能覆盖全部风险。经验攻击未发现泄漏，只说明在给定攻击与样本下没有观察到，不等同于形式保证。

差分隐私比较相邻数据集上的输出分布；它的保证依赖邻接定义、$(\varepsilon,\delta)$、隐私会计、采样假设和实现。DP 合成数据仍可能低效用、含偏差或被不当发布，普通“加噪/抽样/改写”也不能自称 DP。NIST SP 800-226 是评估这类声明及实现隐患的当前权威指导之一，但具体部署仍需专业审查。

## 分步骤方法

1. 列出生成器接触的所有真实数据、提示、检索库和人工示例，按敏感等级分类。
2. 在生成前最小化输入；用虚构标识符，不把真实凭据或个人信息放入提示。
3. 对输出做精确/近似训练样本匹配、稀有组合审查和敏感模式扫描。
4. 若声称差分隐私，记录邻接定义、隐私参数、会计、实现版本和独立审查；不把“加噪”当 DP 同义词。
5. 按群体、语言和交叉层检查覆盖、错误和拒绝率；偏差审查需要相关领域与受影响方参与。
6. 核对所有输入数据、模型服务和输出的许可/合同/政策；不默认“生成后版权归零”。
7. 对高风险或公开发布设置访问控制、Data Card、审批、删除/响应流程和持续监测。

版权与合同问题不能由相似度阈值解决。工程侧应保存来源类别、授权依据、模型/服务条款版本、是否允许衍生与再发布、过滤和人工决定；法律结论交给合格人员。若权利边界不清，限制访问、暂停发布并记录待决问题，不要把“模型生成”当作来源字段的终点。

## 来源—派生产物图与撤回准备

风险审查必须能落到具体工件。为每个输入来源或受控提示记录稳定的 `source_lineage_id`、版本、访问范围、处理目的、证据引用和保留类别；为每次生成记录 `generation_run_id`；再把它们连接到候选、过滤决定、family/split、发布包、索引/缓存和下游评测报告。这个图帮助发现一次来源更正、权限变化或删除请求影响了哪些版本；它本身不裁决权利是否有效。

“当前没有观察到真实数据”与“来源、权利和隐私已审查”是不同状态。前者可作为虚构教学 fixture 的事实声明；后者需要与发布范围对应的受控证据。若 lineage 缺失、来源不可追溯或审查状态待决，应隔离候选或阻止更广范围的发布，不能借助哈希、去重或 PII 正则把它升级为可公开数据。

这里的 `source_lineage_id` 是跨数据工程的抽象引用，不替代 RAG 对租户、文档、source sequence、revision、generation、ACL 与 tombstone 的精确身份合同。合成 case 若引用真实知识来源，应保存到该具体身份的可验证映射，而不是只写一个自由文本 URL；详见[[RAG/09-项目-从来源到引用证据链|RAG 来源到引用的离线证据链]]。

## 示例

风险声明应具体：

~~~text
已证实：候选均来自作者设计的虚构模板；未使用真实客户记录。
已检查：规范化精确重复、常见个人信息模式、条件层覆盖。
未证明：对任意外部语料不存在语义相似；不声称差分隐私。
工程建议：若未来接入真实日志，重新做授权、最小化、隐私与发布审查。
~~~

NIST 明确指出，许多普通合成技术并不满足差分隐私或其他形式化隐私性质。NIST SP 800-226 提供评估 DP 保证及常见实现隐患的指导；它不能替代针对具体系统的专业审计。

## 常见错误与排查

- **synthetic 即匿名**：先追踪真实输入与记忆/链接风险。
- **删除姓名就完成隐私处理**：准标识符组合仍可能关联个人。
- **模型拒绝生成敏感内容就称安全**：检查提示日志、输出、缓存和下游访问。
- **只看平均偏差指标**：检查小样本与交叉群体，记录不确定性。
- **把开源可下载等同于可任意训练/再发布**：核对具体许可证、来源条款和组织政策。
- **只删除最终 JSON 就称完成撤回**：用来源—派生产物图定位数据包、索引、缓存、训练/评测输入与历史报告，并按发布政策处理。

## 自测

1. 输出没有姓名是否证明没有个人信息？不能。
2. 经验攻击未成功是否等于形式隐私保证？不是。
3. generator 的使用条款允许调用，是否自动允许发布其输出数据集？不能仅凭这一事实判断，需核对完整条款、输入权利和用途。

## 小结与下一步

合成是数据处理链的一部分，不会清除上游权利与风险。完成风险审查后，进入 [[数据合成/02-方法与质量/07-版本评测与发布|版本、评测与发布]]，把证据和限制固化到可维护工件中。
