# 克隆风险、披露与可追溯性

## 本节目标

识别冒充、欺诈和未经授权克隆风险，设计声音许可、使用披露、来源记录、检测边界和事件响应。

## 风险不只来自模型

声音克隆可能被用于冒充亲友、领导、客服或公众人物，也可能绕过依赖声音的人工判断。风险链包括参考音频获取、模型/声音创建、脚本生成、分发渠道和受害者处置。仅在音频末尾加一句免责声明无法覆盖实时诈骗、被剪掉的片段或在没有音频末尾的流式交互。

## 分层控制

1. **创建前**：验证声音权利、用途、申请者和高风险限制。
2. **生成时**：内容政策、速率限制、敏感场景拦截、审计 ID。
3. **输出时**：适用时做清晰披露，绑定来源、声音和配置记录。
4. **分发时**：渠道验证、异常监控和用户举报。
5. **事件后**：暂停声音、保全审计记录、通知相关方、支持删除与纠正。

水印或来源凭证可作为一层证据，但可能在压缩、重录、裁剪或格式转换后丢失；检测器也有误报和漏报。不能把“未检测到”解释为“真实”，也不能把元数据当作唯一信任根。即使来源记录能绑定某个输出资产，也只支持“此记录与该资产的关联”；它不证明说话内容真实、听者知情、声音属于某人，或授权仍然有效。跨媒体的内容凭证边界见 [[图像生成/02-工程与质量/07-安全版权与内容凭证|图像生成的安全、版权与内容凭证]]。

## 最小可追溯记录

记录本地 `operation_id`、`source_revision`、声音目录/版本、授权引用、用途、`acl_reference`、对象级授权决定/时间、模型与配置版本、生成时间、操作者/服务身份、输出摘要、披露策略、`release_id` 和后续撤销状态。`acl_reference` 只能指向待核验对象，不能代替授权决定本身。若真实调用供应商，再单独记录 provider 名称、`provider_request_id` 与响应/receipt；它是外部追踪证据，不能替代本地幂等、去重和计划项关联。敏感原文不应为了审计无限期保留；摘要和受控证据要符合保留政策。

把记录分成两类：

- **evidence-bound（已绑定）**：系统实际检查并绑定到该请求/产物的 ID、版本、策略决定、权限主体和完整性证据。
- **evidence-supported（可支持）**：外部合同、同意书、人工复核或渠道回执所支持的事实；应记录其引用、有效期和核验者，但不能只凭字段存在就当作已验证。

这一区分让“有授权引用”与“已核验授权仍有效”不会在日志里被混淆。

## 练习与自测

- 为“公司培训旁白”和“模拟 CEO 语音”分别做风险分级；后者即使内部使用也需严格授权与防外泄。
- 设计一个用户报告疑似冒充音频后的 6 步响应流程：接收与保全最小证据、暂停声音/渠道、验证影响范围、通知与替代、删除/撤销或申诉、复盘与防复发。
- 解释为什么声音相似度高不能证明获得授权。
